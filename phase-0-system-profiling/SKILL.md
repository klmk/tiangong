# Phase 0: 系统画像（System Profiling）

> **阶段定位**：客开的第一步，不是从零定义项目，而是**理解现有系统 + 评估AI能力 + 评估客开可行性**。

---

## 概述

| 属性 | 说明 |
|------|------|
| **阶段名称** | 系统画像（System Profiling） |
| **阶段编号** | Phase 0 |
| **核心目标** | 全面理解目标系统，评估AI辅助客开的可行性与风险 |
| **阶段性质** | 分析与评估阶段，不产出任何代码变更 |

### 与从零开发的区别

| 维度 | 从零开发 | 客开（Customization） |
|------|----------|----------------------|
| 起点 | 空白项目 | 已有系统 |
| 首要任务 | 需求收集 | 系统理解 |
| 风险来源 | 需求不明确 | 对现有系统理解不足 |
| 约束条件 | 技术选型自由 | 受限于现有技术栈和架构 |
| AI角色 | 从零生成代码 | 理解并修改已有代码 |

---

## 入口条件

- [x] 已有目标系统（代码库 / 文档 / 部署环境）
- [x] 已明确客开的目标方向（至少有初步的变更意向）
- [x] 已获得目标系统的访问权限（代码读取、文档查阅、环境访问）
- [x] 已确认与系统相关的关键干系人（使用者、维护者、决策者）

---

## 7步执行流程

### Step 1: 系统信息收集

**目标**：收集目标系统的基本技术信息，建立初步认知。

**执行要点**：
1. **技术栈识别**
   - 前端框架及版本（React / Vue / Angular 等）
   - 后端框架及版本（Spring Boot / Django / Express 等）
   - 数据库类型及版本（MySQL / PostgreSQL / Oracle 等）
   - 中间件及基础设施（Redis / Kafka / Nginx 等）
2. **架构风格判断**
   - 单体 / 微服务 / SOA / Serverless
   - 分层架构 / MVC / CQRS / 事件驱动
3. **代码规模评估**
   - 代码行数（按语言分类）
   - 模块/包数量
   - 核心业务代码占比
4. **部署方式确认**
   - 物理机 / 虚拟机 / 容器化（Docker/K8s）
   - 云服务 / 私有化部署
   - CI/CD 流程

**输出**：系统基本信息表（填入 `system-profile-template.md` 第1章）

---

### Step 2: 代码库结构分析

**目标**：深入理解代码库的组织方式和模块关系。

**执行要点**：
1. **目录结构梳理**
   - 顶层目录划分及职责
   - 源码 / 配置 / 测试 / 文档 / 脚本的分布
   - 多模块项目的模块划分逻辑
2. **模块划分识别**
   - 业务模块 vs 基础设施模块 vs 公共模块
   - 模块间的调用关系
   - 模块的职责边界是否清晰
3. **依赖关系分析**
   - 内部模块依赖图
   - 外部第三方依赖清单及版本
   - 依赖的健康度（是否活跃维护、是否有已知漏洞）
4. **关键入口识别**
   - API 入口（Controller / Router）
   - 定时任务入口
   - 消息消费入口
   - 数据库迁移入口

**输出**：技术画像 + 架构画像（填入 `system-profile-template.md` 第2、5章）

---

### Step 3: 系统画像生成

**目标**：基于前两步的信息，生成完整的系统画像文档。

**执行要点**：
1. **技术画像**：整合技术栈、架构、依赖信息
2. **业务画像**：通过代码和文档推断核心业务流程
   - 识别核心实体和数据模型
   - 梳理主要业务流程
   - 识别用户角色和权限模型
3. **质量画像**：评估代码质量和系统健康度
   - 代码规范程度
   - 测试覆盖情况
   - 文档完整度
   - 技术债务水平
   - 已知问题和历史Bug模式

**输出**：完整系统画像文档（`system-profile-template.md`）

---

### Step 4: AI能力画像

**目标**：评估AI在当前系统上的理解和修改能力。

**执行要点**：
1. **代码理解能力评估**
   - 大型代码库理解：能否在上下文窗口内理解关键模块
   - 跨模块依赖分析：能否追踪跨模块的调用链路
   - 业务逻辑推断：能否从代码推断出业务含义
   - 遗留代码适配：能否理解不规范/过时的代码风格
2. **变更影响评估能力**
   - 能否准确评估一处修改的波及范围
   - 能否识别隐式依赖（如通过数据库、配置文件、事件）
3. **回归风险识别能力**
   - 能否识别修改可能破坏的现有功能
   - 能否建议必要的回归测试范围
4. **代码生成适配能力**
   - 能否按照现有代码风格生成新代码
   - 能否遵循现有的架构模式和设计规范

**输出**：AI能力画像（`ai-capability-profile-template.md`）

---

### Step 5: 客开可行性评估

**目标**：综合系统画像和AI能力画像，评估客开的整体可行性。

**执行要点**：
1. **技术可行性分析**
   - 技术栈兼容性：AI是否熟悉目标技术栈
   - 架构适配度：现有架构是否易于扩展和修改
   - 依赖风险：第三方依赖是否构成障碍
   - 环境要求：开发、测试、部署环境是否满足
2. **业务风险评估**
   - 影响范围：客开变更涉及的业务面
   - 业务连续性：变更期间系统是否需要持续运行
   - 数据安全：是否涉及敏感数据处理
   - 回滚方案：变更失败时的回退策略
3. **AI能力匹配度**
   - 需求-AI能力匹配矩阵
   - 识别AI能力不足的风险点
   - 制定补强方案（如人工介入点）

**输出**：客开可行性报告（`feasibility-assessment-template.md`）

---

### Step 6: 对齐策略

**目标**：与使用者、维护者对齐理解，建立协作规则。

**执行要点**：
1. **利益相关者角色识别**
   - 识别所有参与客开的利益相关者及其角色
   - 按角色分类：
     - **决策者**：对方案有最终决定权（如项目负责人、产品经理）
     - **技术评审者**：评估技术方案可行性（如技术负责人、架构师）
     - **业务验证者**：验证功能是否符合业务预期（如产品经理、业务方）
     - **使用者**：系统的最终使用者
   - 为每个角色确定沟通方式和报告详细程度：
     - **非技术角色**（产品经理、业务方）：关注功能影响、风险点、业务稳定性，不需要技术细节
     - **技术角色**（技术负责人、开发者）：关注技术方案、兼容性、代码质量，需要完整技术细节
   - 记录角色信息到系统画像文档
2. **沟通协议建立**
   - 确定沟通渠道和频率
   - 确定信息同步方式（日报/周报/里程碑报告）
   - 确定决策流程和审批机制
3. **风险缓解策略**
   - 针对已识别风险的缓解措施
   - 应急预案和升级机制
4. **约束识别与确认**
   - 技术约束（不可修改的模块、必须兼容的接口）
   - 业务约束（业务连续性要求、数据一致性要求）
   - 时间约束（里程碑节点、上线窗口）
   - 资源约束（可用的人力、测试环境）
5. **协作规范约定**
   - 代码提交规范
   - 代码审查流程
   - 测试验收标准

**输出**：对齐策略文档（作为可行性报告的一部分或独立文档）

---

### Step 7: 进入下一阶段

**目标**：确认所有准备工作就绪，通过门控检查，进入变更需求分析阶段。

**执行要点**：
1. 逐项检查 Gate 0 出口条件
2. 确认所有必输出物已完成并经过审核
3. 与使用者确认系统画像的准确性
4. 与使用者确认可行性评估的结论
5. 获得进入 Phase 1 的正式批准

**输出**：Gate 0 通过确认

---

## 5条铁律

### 铁律一：先理解后动手
> 在充分理解系统之前，不进行任何代码修改。理解是客开的基础，不理解就动手是最大的风险来源。

### 铁律二：不理解的代码不改
> 如果无法理解某段代码的作用和影响范围，必须先搞清楚再修改。宁可多花时间理解，也不要盲目修改。

### 铁律三：画像不完整不开工
> 系统画像必须覆盖所有关键维度。画像不完整意味着理解有盲区，盲区就是风险。宁可画像过度详细，也不要遗漏关键信息。

### 铁律四：可行性不确认不承诺
> 在可行性评估未完成并通过确认之前，不对客开结果做任何承诺。包括工期、成本、效果。

### 铁律五：约束先于方案
> 任何客开方案都必须在系统约束的框架内制定。约束识别是方案设计的前置条件，不是事后补充。

## 出口条件（Gate 0 - 硬性门控）

以下所有条件必须全部满足，方可进入 Phase 1：

| 序号 | 门控项 | 检查标准 | 状态 |
|------|--------|----------|------|
| G0-1 | 系统画像完整性 | 7个章节全部填写，无空白关键项 | [ ] |
| G0-2 | 技术画像准确性 | 使用者确认技术栈、架构描述准确 | [ ] |
| G0-3 | 业务画像准确性 | 使用者确认核心业务流程描述准确 | [ ] |
| G0-4 | AI能力评估完成 | 4个能力维度全部评估，有明确评分 | [ ] |
| G0-5 | 可行性等级确定 | 可行性评估结论明确（高/中/低/不建议） | [ ] |
| G0-6 | 关键风险已识别 | 所有高风险项已识别并有缓解措施 | [ ] |
| G0-7 | 约束已确认 | 技术约束、业务约束、时间约束已全部确认 | [ ] |
| G0-8 | 利益相关者已识别 | 所有利益相关者角色已识别，沟通方式已确定 | [ ] |
| G0-9 | 使用者确认签字 | 使用者已审阅并确认所有输出物 | [ ] |

**如果可行性等级为"低"或"不建议"**：需要重新评估客开方案或调整客开范围，不应直接进入 Phase 1。

## 红旗列表

以下情况出现时，必须立即暂停并升级处理：

| 序号 | 红旗 | 说明 | 处理方式 |
|------|------|------|----------|
| R0-1 | 代码库无法访问 | 无法获取完整代码或关键模块代码 | 暂停，协调访问权限 |
| R0-2 | 技术栈完全陌生 | AI对目标技术栈无任何知识储备 | 暂停，评估是否需要额外学习或人工介入 |
| R0-3 | 架构文档完全缺失 | 无任何架构文档且代码无注释 | 暂停，评估逆向工程成本 |
| R0-4 | 核心模块不可修改 | 客开目标涉及不可修改的核心模块 | 暂停，重新评估客开范围 |
| R0-5 | 依赖严重过时 | 核心依赖已停止维护且存在安全漏洞 | 暂停，评估升级风险 |
| R0-6 | 数据安全风险 | 客开涉及敏感数据且无合规方案 | 暂停，必须先解决合规问题 |
| R0-7 | 无测试覆盖 | 系统零测试覆盖且无法建立回归测试 | 暂停，评估是否需要先建立测试基线 |
| R0-8 | 使用者无法配合 | 关键干系人无法提供必要的信息和支持 | 暂停，协调资源 |
| R0-9 | 系统处于不稳定状态 | 目标系统存在严重Bug或性能问题 | 暂停，建议先修复稳定系统 |
| R0-10 | 知识产权/许可限制 | 代码许可不允许修改或存在法律风险 | 暂停，必须先解决法律问题 |
