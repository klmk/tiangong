# 代码审查指南（客开版）

> 本指南在 Paoding 代码审查标准的基础上，增加了客开（定制化开发）场景特有的审查维度。
> 客开场景的核心挑战：**在修改已有系统时，确保不破坏已有功能、不偏离现有架构。**

---

## 1. 客开特有的审查维度

### 1.1 变更影响审查

> 审查变更是否超出了变更方案定义的范围。

#### 审查要点

| # | 检查项 | 审查标准 | 严重程度 |
|---|--------|----------|----------|
| CI-1 | 变更是否与变更方案一致 | 每处代码修改都能在变更方案中找到对应的需求 | 阻塞 |
| CI-2 | 是否存在超出方案的额外修改 | 不允许出现方案之外的代码修改（格式化除外） | 阻塞 |
| CI-3 | 变更是否影响了方案中未标注的模块 | 如果影响了未标注的模块，需要补充影响分析 | 阻塞 |
| CI-4 | 变更是否引入了方案中未提及的新依赖 | 新依赖必须在方案中有说明和理由 | 阻塞 |
| CI-5 | 变更是否改变了方案中未提及的接口 | 接口变更必须在方案中有说明 | 阻塞 |
| CI-6 | 变更是否修改了配置文件 | 配置变更必须在方案中有说明 | 重要 |
| CI-7 | 变更是否涉及数据库操作 | 数据库变更必须在方案中有详细的迁移方案 | 阻塞 |

#### 审查方法

```
Step 1: 对比变更代码与变更方案
        ├── 每处代码修改 → 查找对应的变更需求（CHG-ID）
        └── 每个变更需求 → 查找对应的代码修改

Step 2: 检查是否有"无主"的代码修改
        ├── 发现无主修改 → 阻塞，要求说明或回退
        └── 所有修改都有对应需求 → 通过

Step 3: 检查是否有遗漏的修改
        ├── 变更方案中要求修改但未修改 → 阻塞，要求补充
        └── 所有方案要求都已实现 → 通过
```

### 1.2 回归风险审查

> 审查变更是否可能破坏已有功能。

#### 审查要点

| # | 检查项 | 审查标准 | 严重程度 |
|---|--------|----------|----------|
| RR-1 | 是否为受影响的已有功能编写了回归测试 | 所有受影响功能必须有回归测试保护 | 阻塞 |
| RR-2 | 回归测试是否在变更前建立了基线 | 必须有基线通过的记录 | 阻塞 |
| RR-3 | 回归测试是否覆盖了间接影响 | 间接影响的功能也应有回归测试 | 重要 |
| RR-4 | 变更是否修改了被广泛调用的公共函数 | 公共函数的修改需要更严格的审查 | 阻塞 |
| RR-5 | 变更是否修改了数据结构 | 数据结构变更需要评估对所有使用者的影响 | 阻塞 |
| RR-6 | 变更是否修改了全局状态或配置 | 全局状态变更需要评估对并发场景的影响 | 重要 |
| RR-7 | 变更是否修改了错误处理逻辑 | 错误处理变更可能导致异常行为变化 | 重要 |
| RR-8 | 回归测试是否全部通过 | 所有回归测试必须通过 | 阻塞 |

#### 高风险变更模式

以下变更模式在客开场景中风险较高，需要特别关注：

| 变更模式 | 风险描述 | 审查重点 |
|----------|----------|----------|
| 修改公共接口签名 | 可能破坏所有调用方 | 检查所有调用方是否已适配 |
| 修改数据模型 | 可能影响数据持久化和查询 | 检查数据迁移和兼容性 |
| 修改枚举/常量值 | 可能影响依赖这些值的逻辑 | 检查所有引用点 |
| 修改权限/认证逻辑 | 可能影响安全性和访问控制 | 检查所有受保护的资源 |
| 修改异步/并发逻辑 | 可能引入竞态条件 | 检查并发场景下的行为 |
| 修改缓存策略 | 可能导致数据不一致 | 检查缓存失效和更新逻辑 |
| 修改日志/监控 | 可能影响问题排查 | 检查关键日志是否保留 |

### 1.3 兼容性审查

> 审查变更是否与现有架构兼容。

#### 审查要点

| # | 检查项 | 审查标准 | 严重程度 |
|---|--------|----------|----------|
| CC-1 | 变更是否符合现有架构的分层原则 | 不允许跨层直接调用（除非架构本身如此） | 阻塞 |
| CC-2 | 变更是否符合现有架构的依赖方向 | 不允许反向依赖（如业务层依赖展示层） | 阻塞 |
| CC-3 | 变更是否遵循了现有架构的设计模式 | 新代码应使用与现有代码一致的设计模式 | 重要 |
| CC-4 | 变更是否与现有架构的技术选型一致 | 不允许引入与现有技术栈冲突的技术 | 重要 |
| CC-5 | 变更是否遵循了现有的编码规范 | 代码风格必须与现有代码保持一致 | 重要 |
| CC-6 | 变更是否考虑了现有架构的约束 | 如性能约束、安全约束、部署约束 | 重要 |

#### 兼容性审查清单

```
架构兼容性检查：
├── 分层原则：[通过/不通过] - [说明]
├── 依赖方向：[通过/不通过] - [说明]
├── 设计模式：[通过/不通过] - [说明]
├── 技术选型：[通过/不通过] - [说明]
├── 编码规范：[通过/不通过] - [说明]
└── 架构约束：[通过/不通过] - [说明]
```

### 1.4 向后兼容审查

> 审查接口变更是否向后兼容。

#### 审查要点

| # | 检查项 | 审查标准 | 严重程度 |
|---|--------|----------|----------|
| BC-1 | 新增接口参数是否有默认值 | 新增的必填参数会破坏现有调用方 | 阻塞 |
| BC-2 | 修改返回值类型是否兼容 | 返回值类型变更可能导致调用方类型错误 | 阻塞 |
| BC-3 | 删除接口是否有过渡方案 | 直接删除接口会破坏所有调用方 | 阻塞 |
| BC-4 | 修改接口行为是否有版本控制 | 行为变更应通过版本号区分 | 重要 |
| BC-5 | 数据格式变更是否有迁移方案 | 数据格式变更可能导致旧数据无法解析 | 阻塞 |
| BC-6 | 配置项变更是否有兼容处理 | 配置变更不应导致系统无法启动 | 重要 |

#### 向后兼容变更策略

| 变更类型 | 兼容策略 | 示例 |
|----------|----------|------|
| 新增参数 | 提供默认值，保持旧调用方式可用 | `function(name, options = {})` |
| 修改返回值 | 新增字段而非修改/删除已有字段 | 返回对象新增 `newField`，保留 `oldField` |
| 删除接口 | 先标记为 `@Deprecated`，保留至少一个版本周期 | `@Deprecated` 注解 + 文档说明 |
| 修改行为 | 通过配置开关控制新旧行为 | `feature.new_behavior = true` |
| 数据格式 | 支持新旧两种格式的解析 | 版本号区分：`{"version": 2, ...}` |

## 2. 通用代码审查标准

> 以下审查标准适用于所有场景，包括客开和从零开发。

### 2.1 代码正确性

| # | 检查项 | 审查标准 | 严重程度 |
|---|--------|----------|----------|
| COR-1 | 代码逻辑是否正确 | 代码实现了预期的功能，无逻辑错误 | 阻塞 |
| COR-2 | 边界条件是否处理 | 空值、零值、最大值等边界条件有正确处理 | 阻塞 |
| COR-3 | 异常处理是否完善 | 所有可预见的异常都有处理逻辑 | 阻塞 |
| COR-4 | 资源是否正确释放 | 文件、连接、锁等资源在使用后正确释放 | 阻塞 |
| COR-5 | 并发安全 | 共享状态在并发场景下是安全的 | 阻塞 |
| COR-6 | 数据一致性 | 数据操作满足一致性要求 | 阻塞 |

### 2.2 代码可读性

| # | 检查项 | 审查标准 | 严重程度 |
|---|--------|----------|----------|
| READ-1 | 命名是否清晰 | 变量、函数、类的命名能准确表达其用途 | 重要 |
| READ-2 | 注释是否充分 | 复杂逻辑有必要注释，特别是"为什么"而非"是什么" | 重要 |
| READ-3 | 函数长度是否合理 | 单个函数不超过 50 行（建议） | 建议 |
| READ-4 | 嵌套深度是否合理 | 嵌套不超过 3 层（建议） | 建议 |
| READ-5 | 代码结构是否清晰 | 代码组织有逻辑性，易于理解 | 重要 |

### 2.3 代码可维护性

| # | 检查项 | 审查标准 | 严重程度 |
|---|--------|----------|----------|
| MAIN-1 | 是否有代码重复 | 重复代码应提取为公共函数/方法 | 重要 |
| MAIN-2 | 是否有硬编码 | 魔法数字、硬编码字符串应提取为常量或配置 | 重要 |
| MAIN-3 | 是否有过度耦合 | 模块间的耦合度应尽可能低 | 重要 |
| MAIN-4 | 是否遵循 SOLID 原则 | 代码设计应遵循 SOLID 原则（在合理范围内） | 建议 |
| MAIN-5 | 是否有适当的抽象 | 重复的模式应有适当的抽象 | 建议 |

### 2.4 测试充分性

| # | 检查项 | 审查标准 | 严重程度 |
|---|--------|----------|----------|
| TEST-1 | 是否有单元测试 | 核心逻辑应有单元测试覆盖 | 阻塞（客开场景） |
| TEST-2 | 测试是否覆盖正常/边界/异常路径 | 三层覆盖 | 阻塞（客开场景） |
| TEST-3 | 测试是否有意义 | 测试应验证行为，而非实现细节 | 重要 |
| TEST-4 | 测试是否可维护 | 测试代码质量应与生产代码同等要求 | 重要 |
| TEST-5 | Mock 使用是否合理 | 不应过度 Mock，导致测试失去意义 | 建议 |

### 2.5 性能影响

| # | 检查项 | 审查标准 | 严重程度 |
|---|--------|----------|----------|
| PERF-1 | 是否引入了 N+1 查询 | 数据库查询应避免 N+1 问题 | 阻塞 |
| PERF-2 | 是否有不必要的循环嵌套 | 嵌套循环的时间复杂度应可接受 | 重要 |
| PERF-3 | 是否有内存泄漏风险 | 长生命周期对象不应持有短生命周期对象的引用 | 重要 |
| PERF-4 | 是否有大量数据的全量加载 | 大数据集应使用分页或流式处理 | 重要 |
| PERF-5 | 缓存使用是否合理 | 缓存应有合理的过期策略和失效机制 | 建议 |

## 3. 审查流程

### 3.1 两阶段审查流程

```
┌─────────────────────────────────────────────────────┐
│                 客开代码审查流程                       │
├─────────────────────────────────────────────────────┤
│                                                     │
│  第一阶段：变更合规审查                               │
│  ├── 变更影响审查（是否超出范围）                     │
│  ├── 回归风险审查（是否可能破坏已有功能）             │
│  ├── 兼容性审查（是否与现有架构兼容）                 │
│  └── 向后兼容审查（接口变更是否向后兼容）             │
│                                                     │
│  ├── 第一阶段通过 → 进入第二阶段                      │
│  └── 第一阶段不通过 → 阻塞，要求修改                  │
│                                                     │
│  第二阶段：代码质量审查                               │
│  ├── 代码正确性                                     │
│  ├── 代码可读性                                     │
│  ├── 代码可维护性                                   │
│  ├── 测试充分性                                     │
│  └── 性能影响                                       │
│                                                     │
│  ├── 第二阶段通过 → 审查完成                          │
│  └── 第二阶段不通过 → 根据严重程度决定                │
│      ├── 阻塞问题 → 必须修复                         │
│      ├── 重要问题 → 应该修复                         │
│      └── 建议问题 → 记录，后续处理                   │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 3.2 审查意见分级

| 级别 | 含义 | 处理方式 | 阻塞继续？ |
|------|------|----------|-----------|
| **阻塞（Blocker）** | 必须修复的严重问题 | 修复后重新审查 | 是 |
| **重要（Major）** | 应该修复的问题 | 修复后确认 | 建议是 |
| **建议（Minor）** | 可以改进的地方 | 记录，后续处理 | 否 |
| **讨论（Discussion）** | 需要讨论的设计决策 | 讨论后决定 | 否 |

## 4. 客开场景的特殊注意事项

### 4.1 遗留代码的审查标准

客开场景中经常需要修改遗留代码。对遗留代码的审查应遵循以下原则：

| 原则 | 说明 |
|------|------|
| **不苛求完美** | 遗留代码可能有很多历史问题，审查应聚焦于变更本身，而非全面重构 |
| **不降低标准** | 虽然不苛求完美，但变更引入的新代码必须符合当前标准 |
| **渐进改善** | 如果变更涉及修改遗留代码，可以在合理范围内改善代码质量 |
| **保持一致性** | 新代码的风格应与周围的遗留代码保持一致（除非遗留代码风格本身有问题） |

### 4.2 "只改不动"的边界

在客开场景中，有时只需要修改配置或数据，而不需要修改代码。这种变更同样需要审查：

| 变更类型 | 审查重点 |
|----------|----------|
| 配置变更 | 配置值是否正确、是否影响其他功能、是否有回滚方案 |
| 数据变更 | 数据是否一致、是否有迁移方案、是否可逆 |
| 资源变更 | 新增/修改的资源（如图片、模板）是否正确引用 |
| 权限变更 | 权限变更是否影响现有用户的访问 |

### 4.3 多人协作的变更审查

当变更涉及多人协作时，审查需要额外关注：

- 变更之间的接口是否一致
- 合并后是否会产生冲突
- 集成后是否需要额外的回归测试

## 5. 审查清单速查

- [ ] 变更是否与变更方案一致

- [ ] 变更是否与现有架构兼容
- [ ] 接口变更是否向后兼容
- [ ] 回归测试是否全部通过

- [ ] 代码逻辑是否正确

- [ ] 命名是否清晰
- [ ] 注释是否充分
- [ ] 测试是否充分
