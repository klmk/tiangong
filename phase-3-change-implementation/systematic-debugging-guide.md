# 系统化调试指南

> 本指南用于 Phase 3（变更实施）中遇到 Bug 或测试失败时的系统化调试。
> 核心原则：**先理解，再定位，后修复**——禁止不理解就修改。

---

## 1. 调试基本原则

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **先理解再修改** | 在没有理解问题根因之前，不允许修改任何代码 |
| **证据驱动** | 每个结论都必须有证据支撑（日志、测试结果、代码分析） |
| **最小复现** | 用最小的步骤复现问题，排除干扰因素 |
| **根因优先** | 优先找到问题的根本原因，而非治标不治本 |
| **变更追溯** | 调试时始终关注"最近的变更是什么" |

### 1.2 禁止行为

| 禁止行为 | 原因 |
|----------|------|
| 不看错误信息就修改代码 | 错误信息通常已经指出了问题方向 |
| 不理解失败原因就尝试修复 | 盲目修改可能引入新的问题 |
| "试试看"式的随机修改 | 随机修改无法保证问题被真正修复 |
| 同时修改多个地方 | 无法确定哪个修改解决了问题 |
| 修改后不验证就继续 | 可能修复了一个问题但引入了另一个 |

## 2. 系统化调试流程

### 2.1 完整调试流程

```
┌─────────────────────────────────────────────────────┐
│                  系统化调试流程                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Step 1: 问题确认                                    │
│  ├── 确认问题的具体表现（错误信息、异常行为）          │
│  ├── 确认问题的复现条件                              │
│  └── 确认问题的严重程度和影响范围                     │
│                                                     │
│  Step 2: 信息收集                                    │
│  ├── 收集错误信息和堆栈跟踪                          │
│  ├── 收集相关日志                                    │
│  ├── 收集测试输出                                    │
│  └── 收集最近的代码变更                              │
│                                                     │
│  Step 3: 假设形成                                    │
│  ├── 基于收集的信息，形成可能的根因假设              │
│  ├── 按可能性排序假设                                │
│  └── 为每个假设设计验证方法                          │
│                                                     │
│  Step 4: 假设验证                                    │
│  ├── 逐一验证假设（从最可能的开始）                  │
│  ├── 每个假设用证据确认或排除                        │
│  └── 如果所有假设都被排除，回到 Step 2 补充信息      │
│                                                     │
│  Step 5: 根因确认                                    │
│  ├── 确认问题的根本原因                              │
│  ├── 确认根因与问题表现的因果关系                    │
│  └── 评估修复的影响范围                              │
│                                                     │
│  Step 6: 修复方案设计                                │
│  ├── 设计修复方案（最小化变更）                      │
│  ├── 评估修复方案的影响                              │
│  └── 确认修复不会引入新的问题                        │
│                                                     │
│  Step 7: 实施修复                                    │
│  ├── 实施修复代码                                    │
│  ├── 编写/更新测试用例                              │
│  └── 运行测试确认修复                                │
│                                                     │
│  Step 8: 回归验证                                    │
│  ├── 运行全部回归测试                                │
│  ├── 确认无新的回归问题                              │
│  └── 记录调试过程和结果                              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 2.2 问题确认

#### 确认问题的具体表现

- **错误信息**：完整的错误消息、错误码、异常类型
- **异常行为**：期望行为 vs 实际行为
- **发生条件**：什么操作/输入/状态下触发
- **发生频率**：每次都发生 / 偶尔发生 / 特定条件下发生
- **影响范围**：影响哪些功能/用户/数据

#### 确认问题的复现条件

| 条件 | 描述 |
|------|------|
| 前置条件 | 复现问题需要满足的前置状态 |
| 触发操作 | 触发问题的具体操作步骤 |
| 输入数据 | 触发问题所需的输入数据 |
| 环境条件 | 特定的环境配置或状态 |

#### 问题确认检查清单

- [ ] 能够稳定复现问题
- [ ] 记录了完整的错误信息
- [ ] 记录了期望行为和实际行为的差异
- [ ] 确认了问题的严重程度
- [ ] 确认了问题的影响范围

## 3. 信息收集策略

### 3.1 错误信息分析

#### 堆栈跟踪分析

```
堆栈跟踪阅读顺序（从下往上）：
1. 最底部的行 → 问题的入口点（通常是你的代码调用的地方）
2. 中间的行 → 调用链（帮助你理解代码路径）
3. 最顶部的行 → 错误发生的位置（异常被抛出的地方）

重点关注：
- 哪些行是"你的代码"（非框架/库代码）
- 异常类型和消息
- 调用链中是否有不期望的路径
```

#### 常见错误信息分类

| 错误类型 | 常见原因 | 调试方向 |
|----------|----------|----------|
| NullPointerException | 对象为空时访问其属性/方法 | 检查对象初始化和空值检查 |
| TypeError | 类型不匹配 | 检查数据类型转换 |
| KeyError/IndexError | 键/索引不存在 | 检查数据结构和边界条件 |
| ConnectionError | 网络/数据库连接失败 | 检查连接配置和网络状态 |
| TimeoutError | 操作超时 | 检查超时配置和性能瓶颈 |
| ConcurrentModificationException | 并发修改共享数据 | 检查并发控制和同步机制 |

### 3.2 日志分析

| 日志级别 | 用途 | 调试时的关注点 |
|----------|------|---------------|
| ERROR | 错误信息 | 错误发生的位置和上下文 |
| WARN | 警告信息 | 可能导致问题的异常情况 |
| INFO | 关键操作记录 | 操作的执行路径和参数 |
| DEBUG | 详细调试信息 | 变量值、方法调用等细节 |

### 3.3 代码变更分析

在客开场景中，问题往往与最近的变更相关：

```
Step 1: 列出最近的代码变更
        git log --oneline -20
        git diff HEAD~5

Step 2: 对比变更与问题
        ├── 问题是否在某个特定变更后出现？
        ├── 变更是否修改了问题相关的代码？
        └── 变更是否修改了问题相关代码的依赖？

Step 3: 使用 Git Bisect 定位引入问题的变更
        git bisect start
        git bisect bad HEAD          # 当前版本有问题
        git bisect good <commit>     # 之前的好版本
        # Git 会自动二分查找引入问题的提交
```

## 4. 假设形成与验证

### 4.1 常见假设模板

| 假设类型 | 模板 | 验证方法 |
|----------|------|----------|
| 数据假设 | "问题是因为 [数据X] 的值为 [意外值]" | 检查/打印数据X的值 |
| 逻辑假设 | "问题是因为 [条件Y] 的判断逻辑有误" | 检查条件Y的判断分支 |
| 时序假设 | "问题是因为 [操作A] 在 [操作B] 之前执行" | 调整执行顺序或添加等待 |
| 状态假设 | "问题是因为 [状态Z] 未被正确重置" | 检查状态Z的生命周期 |
| 依赖假设 | "问题是因为 [依赖D] 的版本/行为变化" | 检查依赖D的版本和行为 |
| 环境假设 | "问题是因为 [环境E] 的配置与预期不同" | 检查环境配置 |

### 4.2 假设验证方法

| 验证方法 | 适用场景 | 操作 |
|----------|----------|------|
| **日志验证** | 需要确认运行时状态 | 在关键位置添加日志，观察输出 |
| **断点调试** | 需要逐步跟踪执行流程 | 使用调试器设置断点，逐步执行 |
| **单元测试** | 需要隔离验证特定逻辑 | 编写针对性的单元测试 |
| **排除法** | 需要缩小问题范围 | 逐步排除可能的因素 |
| **对比法** | 需要确认差异 | 对比正常和异常情况的差异 |
| **回退法** | 怀疑是某个变更引入的 | 回退变更，确认问题是否消失 |

### 4.3 假设验证记录

```
假设验证记录
===========

假设 1：[描述假设]
  验证方法：[描述验证方法]
  验证结果：[确认 / 排除]
  证据：[具体的证据]

假设 2：[描述假设]
  验证方法：[描述验证方法]
  验证结果：[确认 / 排除]
  证据：[具体的证据]

...

根因确认：[最终确认的根因]
  因果链：[根因 → 中间影响 → 最终表现]
```

## 5. 客开特有调试场景

### 5.1 与已有代码的交互问题

客开场景中最常见的调试场景：新代码与已有代码的交互出现问题。

#### 场景描述

```
新代码调用已有代码 → 已有代码的行为与新代码的预期不一致 → 问题发生
```

#### 调试策略

| 步骤 | 操作 | 目的 |
|------|------|------|
| 1 | 理解已有代码的实际行为 | 不要假设已有代码的行为，要实际验证 |
| 2 | 对比已有代码的行为与文档描述 | 已有代码可能与文档不一致 |
| 3 | 检查已有代码的隐式前提条件 | 已有代码可能有未文档化的前提条件 |
| 4 | 检查已有代码的副作用 | 已有代码可能有未预期的副作用 |
| 5 | 检查调用顺序的要求 | 已有代码可能对调用顺序有隐式要求 |

#### 常见交互问题

| 问题类型 | 表现 | 调试方法 |
|----------|------|----------|
| 接口契约不一致 | 新代码期望的返回值与已有代码实际返回不一致 | 检查已有代码的返回值格式 |
| 隐式状态依赖 | 已有代码依赖某个全局状态，新代码未设置 | 检查已有代码的全局状态依赖 |
| 调用时序错误 | 已有代码要求先调用 A 再调用 B，新代码顺序错误 | 检查已有代码的调用时序要求 |
| 隐式数据格式 | 已有代码期望特定的数据格式，新代码提供了不同格式 | 检查已有代码的输入数据格式要求 |
| 未文档化的副作用 | 已有代码在执行主逻辑的同时修改了其他状态 | 检查已有代码的所有写操作 |

### 5.2 数据兼容性问题

客开场景中，数据兼容性问题是另一大类常见调试场景。

#### 场景描述

```
数据结构变更 → 旧数据与新代码不兼容 → 问题发生
```

#### 调试策略

| 步骤 | 操作 | 目的 |
|------|------|------|
| 1 | 检查数据库中的实际数据 | 确认数据是否与预期一致 |
| 2 | 检查数据迁移是否执行成功 | 确认迁移脚本是否正确执行 |
| 3 | 检查缓存中的旧数据 | 缓存中可能存在旧格式的数据 |
| 4 | 检查消息队列中的旧消息 | 队列中可能存在旧格式的消息 |
| 5 | 检查外部系统中的旧数据 | 外部系统可能存储了旧格式的数据 |

#### 常见数据兼容性问题

| 问题类型 | 表现 | 调试方法 |
|----------|------|----------|
| Schema 不匹配 | 新代码期望的字段在旧数据中不存在 | 检查数据库 Schema 和数据 |
| 数据类型不匹配 | 新代码期望的类型与旧数据的类型不同 | 检查字段的实际数据类型 |
| 枚举值变更 | 旧数据中的枚举值在新代码中不再有效 | 检查枚举值的映射关系 |
| 数据格式变更 | JSON/XML 格式发生了变化 | 检查数据的序列化/反序列化 |
| 编码问题 | 字符编码不一致导致数据乱码 | 检查字符编码设置 |

### 5.3 配置兼容性问题

#### 场景描述

```
新代码依赖新的配置项 → 配置缺失或值不正确 → 问题发生
```

#### 调试策略

| 步骤 | 操作 | 目的 |
|------|------|------|
| 1 | 检查新代码依赖的所有配置项 | 确认所有配置项都已设置 |
| 2 | 检查配置项的默认值 | 确认默认值是否合理 |
| 3 | 检查配置项的加载顺序 | 确认配置是否被正确加载 |
| 4 | 检查配置项的覆盖关系 | 确认环境变量是否覆盖了配置文件 |
| 5 | 检查配置项的作用域 | 确认配置是否在正确的作用域内生效 |

## 6. 回归 Bug 的调试策略

### 6.1 区分新引入 Bug vs 已有 Bug

在客开场景中，调试时首先需要区分问题是新引入的还是已存在的。

#### 判断方法

```
Step 1: 在变更前的代码上运行相同的测试
        ├── 测试通过 → Bug 是新引入的，与本次变更相关
        └── 测试失败 → Bug 是已存在的，与本次变更无关

Step 2: 如果是已存在的 Bug
        ├── 检查回归测试基线中是否已记录
        ├── 评估 Bug 的严重程度
        └── 决定是否需要在本次变更中修复

Step 3: 如果是新引入的 Bug
        ├── 使用 Git Bisect 定位引入 Bug 的具体变更
        ├── 分析变更与 Bug 的因果关系
        └── 设计修复方案
```

#### 决策树

```
发现测试失败
├── 回归测试基线中已存在（基线时已失败）
│   ├── 已记录的已知问题 → 标注，继续变更
│   └── 未记录 → 评估是否需要在变更前修复
└── 回归测试基线中不存在（基线时通过）
    ├── 本次变更引入 → 必须修复
    └── 环境变化引入 → 修复环境
```

### 6.2 新引入 Bug 的调试

#### 调试步骤

1. **确认 Bug 是新引入的**：在变更前的代码上验证
2. **定位引入 Bug 的具体变更**：使用 Git Bisect 或二分法
3. **分析变更与 Bug 的因果关系**：
   - 变更修改了什么
   - 修改如何导致了 Bug
   - 是否有遗漏的适配修改
4. **设计修复方案**：
   - 最小化修复
   - 评估修复的影响
   - 确认修复不会引入新的问题
5. **实施修复并验证**：
   - 修复代码
   - 补充测试（防止同类问题再次出现）
   - 运行全部回归测试

### 6.3 已有 Bug 的处理策略

| Bug 严重程度 | 处理策略 |
|-------------|----------|
| **严重**（影响核心功能） | 必须在变更前修复，否则变更可能在此基础上引入更严重的问题 |
| **重要**（影响重要功能） | 评估是否与本次变更的模块相关，如果相关则建议修复 |
| **一般**（影响一般功能） | 记录问题，在变更日志中标注，后续单独修复 |
| **低**（影响较小） | 记录问题，不阻塞变更流程 |

## 8. 常见调试反模式

| 反模式 | 描述 | 正确做法 |
|--------|------|----------|
| **随机修改** | 不分析就随机修改代码试试看 | 先分析根因，再针对性修复 |
| **过度依赖调试器** | 只用调试器单步执行，不分析代码逻辑 | 先阅读代码理解逻辑，再用调试器验证 |
| **忽略错误信息** | 不仔细阅读错误信息就开始调试 | 先完整阅读和分析错误信息 |
| **只看表面** | 只关注错误发生的位置，不追踪调用链 | 从错误位置向上追踪完整的调用链 |
| **修改后不验证** | 修改代码后不运行测试就认为已修复 | 修改后必须运行测试验证 |
| **修复一个引入三个** | 修复一个 Bug 但引入了更多 Bug | 修复后运行全部回归测试 |
| **不记录调试过程** | 调试完成后不记录过程和结论 | 记录完整的调试过程，便于后续参考 |

## 9. 调试工具速查

### 9.1 通用调试命令

```bash
# Git: 查看最近的变更
git log --oneline -20
git diff HEAD~5

# Git: 二分查找引入问题的变更
git bisect start
git bisect bad HEAD
git bisect good <commit>

# Git: 查看特定文件的变更历史
git log --follow -p <file>

# 搜索代码中的相关逻辑
grep -rn "keyword" src/

# 查看进程状态
ps aux | grep <process>

# 查看端口占用
netstat -tlnp | grep <port>

# 查看日志
tail -f /var/log/app.log
grep "ERROR" /var/log/app.log | tail -50
```

### 9.2 测试调试命令

```bash
# 运行单个测试（带详细输出）
pytest tests/test_specific.py::test_function -v -s

# 运行测试并显示打印输出
pytest tests/ -s

# 运行上次失败的测试
pytest --lf

# 运行测试并进入调试器
pytest --pdb tests/test_specific.py

# 运行测试并生成覆盖率报告
pytest tests/ --cov=src --cov-report=term-missing
```
