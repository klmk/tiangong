# 回归测试指南

> 本指南用于 Phase 3（变更实施）中编写和执行回归测试。
> 核心理念：**先保护，再修改**——在变更代码之前，先用测试为已有功能建立安全网。

---

## 1. 回归测试策略

### 1.1 测试范围确定

回归测试范围应基于以下三个来源综合确定：

#### 来源一：Phase 1 的影响分析

- Phase 1 中识别的功能影响范围
- 使用者明确指出的不可破坏的功能
- 业务关键路径上的功能

#### 来源二：Phase 2 的架构分析

- 架构理解文档中的模块依赖关系
- 隐式依赖与耦合点分析结果
- 变更影响热力图中标注的高风险区域

#### 来源三：变更方案的影响标注

- 变更方案中每个变更的直接影响范围
- 变更方案中每个变更的间接影响范围
- 接口变更、数据变更的波及范围

#### 范围确定流程

```
变更需求
├── 直接影响 → 必须覆盖（P0）
├── 直接交互模块 → 应该覆盖（P1）
├── 间接依赖模块 → 建议覆盖（P2）
└── 无明显关联 → 选择性覆盖（P3）
```

### 1.2 测试优先级

| 优先级 | 定义 | 覆盖要求 | 失败处理 |
|--------|------|----------|----------|
| **P0 - 核心** | 变更直接影响的、不可失败的核心功能 | 100% 覆盖，必须编写测试 | 必须修复，阻塞继续 |
| **P1 - 重要** | 与变更模块有直接交互的功能 | 100% 覆盖，必须编写测试 | 必须修复，阻塞继续 |
| **P2 - 一般** | 与变更模块有间接交互的功能 | 重点场景覆盖 | 评估影响，决定是否修复 |
| **P3 - 低** | 受影响概率较低的功能 | 选择性覆盖 | 记录问题，后续处理 |

### 1.3 测试深度

| 测试深度 | 适用场景 | 说明 |
|----------|----------|------|
| **单元测试** | 所有受影响的函数/方法 | 验证单个函数的行为 |
| **集成测试** | 模块间有交互的场景 | 验证模块间的协作 |
| **端到端测试** | 涉及完整业务流程的场景 | 验证端到端的业务流程 |
| **契约测试** | 有接口变更的场景 | 验证接口契约未被破坏 |

## 2. 回归测试编写规范

### 2.1 基线测试

基线测试用于验证已有功能在当前状态下正常工作，作为变更前的参照。

#### 编写原则

1. **基于现有行为，不基于预期行为**：测试应该反映代码当前的实际行为，而不是你认为它"应该"有的行为
2. **覆盖正常路径**：验证功能在正常输入下的输出
3. **覆盖边界条件**：验证功能在边界值下的行为
4. **覆盖异常场景**：验证功能在异常输入下的处理

#### 编写步骤

```
Step 1: 阅读现有代码，理解当前行为
Step 2: 识别关键的输入和输出
Step 3: 编写测试用例，描述当前行为
Step 4: 运行测试，确认通过（GREEN 基线）
Step 5: 如果测试失败，分析原因：
    ├── 测试编写有误 → 修正测试
    ├── 已有Bug → 记录问题，评估处理方式
    └── 代码行为与文档不一致 → 确认实际行为，以代码为准
```

#### 基线测试模板

```python
# 回归基线测试：[功能名称]
# 关联变更：CHG-XXX
# 目的：验证 [功能名称] 在变更前的行为

def test_[功能名称]_正常输入:
    """
    基线测试：验证 [功能名称] 在正常输入下的行为
    当前行为：[描述当前代码的实际行为]
    """
    # Arrange
    input_data = [构造测试输入]

    # Act
    result = [调用被测函数]

    # Assert
    assert result == [期望输出]  # 基于当前实际行为

def test_[功能名称]_边界条件:
    """
    基线测试：验证 [功能名称] 在边界条件下的行为
    当前行为：[描述当前代码在边界条件下的实际行为]
    """
    # Arrange
    input_data = [构造边界值输入]

    # Act
    result = [调用被测函数]

    # Assert
    assert result == [期望输出]  # 基于当前实际行为

def test_[功能名称]_异常输入:
    """
    基线测试：验证 [功能名称] 在异常输入下的行为
    当前行为：[描述当前代码在异常输入下的实际行为]
    """
    # Arrange
    input_data = [构造异常输入]

    # Act & Assert
    with pytest.raises([期望异常类型]):
        [调用被测函数]
```

### 2.2 边界测试

边界测试关注输入和状态的边界条件，这些是变更最容易引入回归Bug的地方。

#### 常见边界条件清单

| 边界类型 | 示例 | 测试要点 |
|----------|------|----------|
| 空值/Null | `null`, `None`, `undefined` | 系统是否正确处理空值 |
| 零值 | `0`, `""`, `[]`, `{}` | 零值是否与"无数据"区分 |
| 最大值 | `Integer.MAX_VALUE`, 数组上限 | 超过最大值时的行为 |
| 最小值 | `Integer.MIN_VALUE`, 负数边界 | 最小值边界的行为 |
| 单元素 | 只有一个元素的集合 | 单元素场景的特殊处理 |
| 空集合 | 空列表、空Map | 空集合的处理逻辑 |
| 并发 | 多线程同时访问 | 并发场景下的数据一致性 |
| 时序 | 操作的先后顺序 | 操作顺序对结果的影响 |

### 2.3 集成测试

集成测试验证模块间的交互是否正常，特别关注变更模块与其他模块的接口。

#### 编写原则

1. **测试真实的模块交互**：不是 mock 其他模块，而是测试真实的集成行为
2. **覆盖关键交互路径**：优先测试数据流和调用链上的关键节点
3. **验证数据一致性**：确保数据在模块间流转时保持一致

#### 集成测试关注点

```
变更模块 A
├── A 调用 B → 测试 A→B 的调用是否正常
├── A 调用 C → 测试 A→C 的调用是否正常
├── D 调用 A → 测试 D→A 的调用是否正常
├── A 写入数据库 → 测试数据写入是否正确
├── A 读取缓存 → 测试缓存读取是否正确
└── A 发布事件 → 测试事件是否正确发布和处理
```

## 3. 变更测试编写规范

### 3.1 定义期望行为

变更测试用于定义变更后的期望行为，是变更是否成功的判定标准。

#### 编写原则

1. **基于变更需求，不基于实现细节**：测试应该描述"期望什么行为"，而不是"怎么实现"
2. **覆盖正常/边界/异常路径**：与回归测试相同的三层覆盖
3. **每个测试关联变更需求**：测试必须标注对应的 CHG-ID

#### 变更测试模板

```python
# 变更测试：[变更标题]
# 关联变更：CHG-XXX
# 目的：验证变更 CHG-XXX 的期望行为

def test_CHG_XXX_[功能名称]_正常行为:
    """
    变更测试：验证 CHG-XXX 在正常输入下的期望行为
    变更需求：[描述变更需求]
    期望行为：[描述变更后的期望行为]
    """
    # Arrange
    input_data = [构造测试输入]

    # Act
    result = [调用被测函数]

    # Assert - 验证变更后的期望行为
    assert result == [变更后的期望输出]

def test_CHG_XXX_[功能名称]_边界行为:
    """
    变更测试：验证 CHG-XXX 在边界条件下的期望行为
    """
    # Arrange
    input_data = [构造边界值输入]

    # Act
    result = [调用被测函数]

    # Assert
    assert result == [变更后的期望输出]

def test_CHG_XXX_[功能名称]_异常行为:
    """
    变更测试：验证 CHG-XXX 在异常情况下的期望行为
    """
    # Arrange
    input_data = [构造异常输入]

    # Act & Assert
    with pytest.raises([期望异常类型]):
        [调用被测函数]
```

### 3.2 覆盖正常路径

- 验证变更后的功能在正常输入下产生正确输出
- 覆盖变更需求中描述的所有功能点
- 覆盖变更需求中描述的所有业务规则

### 3.3 覆盖边界路径

- 变更引入的新参数的边界值
- 变更修改的逻辑分支的边界条件
- 变更新增的数据结构的容量边界

### 3.4 覆盖异常路径

- 变更后的错误处理逻辑
- 变更引入的新异常场景
- 变更后的降级/容错行为

## 4. 测试执行流程

### 4.1 完整流程

```
┌─────────────────────────────────────────────────────┐
│                    Phase 3 测试流程                    │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Step 1: 编写回归测试                                │
│  ├── 基线测试（已有功能的正常/边界/异常路径）          │
│  └── 集成测试（模块间交互）                           │
│                                                     │
│  Step 2: 运行回归测试 → 确认全部通过（GREEN 基线）    │
│  ├── 全部通过 → 基线建立成功                          │
│  └── 存在失败 → 修复后再建立基线                      │
│                                                     │
│  Step 3: 编写变更测试                                │
│  ├── 正常路径测试                                    │
│  ├── 边界路径测试                                    │
│  └── 异常路径测试                                    │
│                                                     │
│  Step 4: 运行变更测试 → 确认失败（RED）               │
│  ├── 测试失败 → 变更测试有效                          │
│  └── 测试通过 → 检查测试是否有效                      │
│                                                     │
│  Step 5: 实施变更                                    │
│  └── 逐步实现，使变更测试通过（GREEN）                │
│                                                     │
│  Step 6: 运行全部测试                                │
│  ├── 回归测试全部通过 → 无破坏                        │
│  └── 回归测试存在失败 → 修复回归Bug                   │
│                                                     │
│  Step 7: 最终验证                                    │
│  └── 全量回归测试 + 变更测试 → 全部通过               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 4.2 基线建立

**目的：** 在变更前建立一个"所有测试都通过"的基准线。

**步骤：**

1. 编写所有回归测试
2. 运行全部回归测试
3. 确认 100% 通过
4. 记录测试结果（测试数量、耗时、覆盖率）
5. 保存测试报告

**基线记录模板：**

```
基线建立记录
===========
日期：[日期]
变更ID：[CHG-XXX]
测试总数：[数量]
通过数量：[数量]
失败数量：0
跳过数量：[数量]
执行耗时：[时间]
覆盖率：[百分比]
测试命令：[完整命令]
测试报告路径：[路径]
```

### 4.3 变更实施中

**频率：** 每完成一个小的代码变更后立即运行相关测试。

**运行范围：**
- 最小范围：与当前变更直接相关的测试
- 推荐范围：当前变更相关的测试 + 直接影响模块的回归测试
- 里程碑范围：全部回归测试

### 4.4 回归验证

**时机：** 变更实施完成后。

**运行范围：** 全量回归测试。

**通过标准：**
- 所有回归测试通过
- 所有变更测试通过
- 无新增的跳过测试
- 测试覆盖率不低于基线

## 5. 回归测试反模式

> 以下是客开场景中常见的回归测试反模式，必须避免。

### 反模式 1：只测改动的函数

**错误做法：** 只为直接修改的函数编写测试，忽略了调用该函数的上层功能和依赖该函数的下层功能。

**正确做法：** 从功能影响范围出发，覆盖所有受影响的功能，包括间接影响。

```
错误：修改了 OrderService.calculateDiscount()，只为这个方法写测试
正确：修改了 OrderService.calculateDiscount()，为以下功能写测试：
  ├── calculateDiscount() 方法本身
  ├── 调用 calculateDiscount() 的 OrderService.createOrder()
  ├── 依赖折扣计算结果的 OrderService.getOrderTotal()
  └── 展示订单总额的前端页面（端到端测试）
```

### 反模式 2：忽略间接影响

**错误做法：** 只测试直接受影响的模块，忽略了通过隐式依赖间接影响的模块。

**正确做法：** 基于架构分析中的隐式依赖和耦合点，覆盖间接影响。

```
错误：修改了数据库表 A 的 Schema，只测试直接操作表 A 的模块
正确：修改了数据库表 A 的 Schema，还测试：
  ├── 通过视图关联表 A 的查询
  ├── 依赖表 A 数据的缓存逻辑
  └── 使用表 A 数据的报表功能
```

### 反模式 3：Mock 掉所有依赖

**错误做法：** 在集成测试中 Mock 掉所有外部依赖，导致测试无法发现真实的集成问题。

**正确做法：** 在单元测试中使用 Mock，但在集成测试中使用真实的依赖。

```
错误：集成测试中 Mock 了数据库，无法发现 SQL 兼容性问题
正确：
  ├── 单元测试：Mock 数据库，测试业务逻辑
  └── 集成测试：使用真实数据库，测试数据访问层
```

### 反模式 4：测试与实现耦合

**错误做法：** 测试代码依赖实现细节（如内部方法名、私有变量），导致重构时测试失败。

**正确做法：** 测试应该基于公开接口和行为契约，不依赖实现细节。

```python
# 错误：测试依赖内部实现
def test_order_processing:
    order._internal_calculate()  # 测试私有方法
    assert order._discount == 0.8  # 测试私有变量

# 正确：测试基于公开接口
def test_order_processing:
    order.apply_discount("VIP2024")  # 测试公开方法
    assert order.get_total() == 800  # 测试公开方法的结果
```

### 反模式 5：基线测试描述期望行为而非当前行为

**错误做法：** 基线测试描述的是"代码应该有的行为"，而不是"代码当前的实际行为"。

**正确做法：** 基线测试必须反映代码当前的真实行为。如果当前行为有Bug，基线测试也应该反映这个Bug（并在注释中标注）。

```python
# 错误：基线测试描述期望行为
def test_calculate_price_baseline:
    result = calculate_price(quantity=0)
    assert result == 0  # 期望行为，但当前代码可能返回 None

# 正确：基线测试描述当前行为
def test_calculate_price_baseline:
    """
    基线测试：当前行为
    注意：当 quantity=0 时，当前代码返回 None（已知问题 #1234）
    """
    result = calculate_price(quantity=0)
    assert result is None  # 当前实际行为
```

### 反模式 6：不运行基线就修改代码

**错误做法：** 编写了回归测试但不运行确认就立即开始修改代码。

**正确做法：** 必须先运行回归测试确认全部通过（GREEN 基线），然后才能开始修改代码。

### 反模式 7：回归测试通过后就不再关注

**错误做法：** 回归测试通过后就认为安全了，不再关注测试的质量和覆盖度。

**正确做法：** 定期审查回归测试的质量，确保覆盖了所有应该覆盖的场景。

## 6. 常见借口及反驳

### 借口 1："这个模块太复杂了，没法写测试"

**反驳：** 正因为复杂，才更需要测试。复杂的模块恰恰是变更最容易引入Bug的地方。如果模块太复杂无法测试，说明模块本身需要重构——但这应该在变更之前或之后单独处理，不能作为跳过测试的理由。

### 借口 2："我只是改了一行代码，不需要回归测试"

**反驳：** 一行代码的修改也可能引发蝴蝶效应。特别是在客开场景中，一行代码可能影响：
- 调用该代码的所有上层功能
- 依赖该代码返回值的所有下游逻辑
- 通过隐式依赖关联的远程模块

### 借口 3："已有的测试已经覆盖了，不需要再写"

**反驳：** 需要确认已有测试是否真的覆盖了本次变更的影响范围。很多时候已有测试覆盖的是"正常路径"，而变更可能影响的是"边界条件"或"异常路径"。此外，已有测试可能已经过时或失效。

### 借口 4："写测试太花时间了，影响进度"

**反驳：** 不写测试的代价更大。在客开场景中，修复一个因为缺少回归测试而遗漏的Bug，其成本远高于提前编写测试的成本。测试不是进度障碍，而是进度保障。

### 借口 5："这个功能是内部使用的，不需要那么严格的测试"

**反驳：** 内部使用的功能同样可能影响外部可见的行为。而且，"内部"和"外部"的边界在客开场景中往往不清晰。一个内部模块的Bug可能通过数据流传播到外部可见的功能。

### 借口 6："测试环境跑不过，先跳过，后面再补"

**反驳：** 测试环境跑不过本身就是红旗。如果测试环境有问题，说明生产环境可能也有类似问题。跳过测试等于在没有安全网的情况下走钢丝。

### 借口 7："这个变更不会影响其他功能"

**反驳：** 除非有充分的架构分析和影响评估作为依据，否则"不会影响"只是一种假设。客开场景中的系统往往充满隐式依赖，"不会影响"的假设经常被证明是错误的。

## 附录

### A. 回归测试检查清单

在提交变更前，逐项确认：

### B. 回归测试命令速查

```bash
# 运行单个测试文件
pytest tests/test_order_service.py -v

# 运行特定标记的测试
pytest tests/ -k "regression" -v

# 运行特定变更相关的测试
pytest tests/ -k "CHG_001" -v

# 运行全量回归测试
pytest tests/ --regression -v

# 运行测试并生成覆盖率报告
pytest tests/ --cov=src --cov-report=html

# 运行测试并生成 JUnit 格式报告
pytest tests/ --junitxml=test-report.xml
```
