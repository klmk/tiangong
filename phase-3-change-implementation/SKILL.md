# Phase 3: 变更实施

> **定位：不是写新代码，而是"先保护再修改"——先写回归测试保护已有功能，再实施变更**

---

## 概述

本阶段是客开流程中将变更方案落地的执行阶段。与从零开发不同，客开的核心挑战在于：**在修改已有系统时，不能破坏已有功能**。

本阶段采用严格的"保护-变更-验证"流程：

1. **先保护**：为受影响的已有功能编写回归测试，建立安全网
2. **再变更**：在测试保护下实施变更
3. **后验证**：运行全量回归测试，确保无破坏

本阶段遵循 TDD（测试驱动开发）的核心理念，但做了客开场景的适配：

- **回归测试（RED → GREEN 基线）**：先为已有功能写测试，确认通过，作为变更前的基线
- **变更测试（RED）**：编写定义变更期望行为的测试，确认失败
- **实施变更（GREEN）**：用最少代码使变更测试通过
- **回归验证（REFACTOR）**：运行全部回归测试，确认无破坏

---

## 入口条件

| 条件 | 说明 |
|------|------|
| Gate 2 通过 | Phase 2（架构理解与变更方案）的硬性门控已通过 |
| 变更方案已确认 | 所有变更方案已通过评审并获得使用者确认 |
| 实施计划已就绪 | 详细的、无占位符的实施计划已制定 |
| 回归测试计划已就绪 | 回归测试范围和策略已明确 |
| 开发环境已就绪 | 代码可编译、可运行、可测试 |

## 执行流程

### Step 1: 任务领取与上下文理解

**目标：充分理解要实施的变更任务及其上下文**

#### 1.1 任务确认

- 从实施计划中领取下一个待执行的任务
- 确认任务对应的变更需求（CHG-ID）
- 确认任务的依赖任务已完成
- 确认任务涉及的所有文件

#### 1.2 上下文加载

- 阅读架构理解文档中与该任务相关的部分
- 阅读变更方案文档中该任务的详细方案
- 阅读涉及文件的现有代码
- 理解该文件在整体架构中的位置和角色

#### 1.3 变更影响确认

- 确认该变更的直接影响范围
- 确认该变更的间接影响范围
- 确认需要保护的已有功能列表
- 确认对应的回归测试命令

---

### Step 2: 回归测试编写（RED → GREEN 基线）

**目标：为受影响的已有功能编写测试，建立变更前的安全基线**

#### 2.1 识别受影响功能

- 基于变更方案的影响范围标注
- 基于架构分析的隐式依赖
- 基于实施计划中的回归测试命令

#### 2.2 编写回归测试

为每个受影响的已有功能编写测试：

- **基线测试**：验证已有功能在当前状态下正常工作
- **边界测试**：覆盖已有功能的边界条件
- **集成测试**：验证模块间交互正常

详细规范参见 `regression-testing-guide.md`。

#### 2.3 建立基线

- 运行所有回归测试，确认全部通过（GREEN）
- 记录基线测试结果（测试报告、截图等）
- 如果回归测试失败，说明已有功能已经存在问题，需要先修复

**关键判断：**

```
回归测试编写完成 → 运行测试
├── 全部通过 → 基线建立成功，进入 Step 3
└── 存在失败 → 分析失败原因
    ├── 已有Bug（非本次变更引入） → 记录问题，评估是否需要在变更前修复
    └── 测试编写错误 → 修正测试
```

---

### Step 3: 变更测试编写（RED）

**目标：编写定义变更期望行为的测试，确认测试失败**

#### 3.1 测试用例设计

基于变更方案，设计测试用例：

- **正常路径**：变更后的功能在正常输入下的行为
- **边界路径**：变更后的功能在边界条件下的行为
- **异常路径**：变更后的功能在异常情况下的行为

#### 3.2 编写测试代码

- 编写测试代码，明确描述变更后的期望行为
- 每个测试用例关联到具体的变更需求（CHG-ID）
- 测试命名清晰，描述期望行为

#### 3.3 确认测试失败（RED）

- 运行变更测试，确认测试失败
- 如果测试通过，说明：
  - 测试编写有误（未真正验证变更行为），或
  - 现有代码已经满足变更需求（需要确认）

---

### Step 4: 实施变更（GREEN）

**目标：用最少的代码使变更测试通过**

#### 4.1 实施原则

- **最小化变更**：只写使测试通过所必需的代码
- **遵循方案**：严格按照变更方案实施，不自行发挥
- **保持风格**：新代码风格与现有代码保持一致
- **可追溯**：每段变更代码关联到变更需求

#### 4.2 实施步骤

1. 按照实施计划中的具体操作步骤执行
2. 每完成一个小的代码变更，立即运行相关测试
3. 逐步推进，直到所有变更测试通过

#### 4.3 变更记录

- 每段变更代码添加注释，标注变更ID和原因
- 使用 Git 提交信息关联变更需求
- 更新变更日志

---

### Step 5: 回归验证（REFACTOR）

**目标：运行全部回归测试，确认变更没有破坏已有功能**

#### 5.1 运行回归测试

- 运行该任务对应的所有回归测试
- 运行该变更影响范围内的全部回归测试
- 如果有全局回归测试套件，运行全量回归

#### 5.2 结果分析

```
回归测试运行完成
├── 全部通过 → 回归验证成功，进入 Step 6
└── 存在失败 → 分析失败原因
    ├── 变更引入的回归Bug → 修复变更代码，重新验证
    ├── 已有Bug（基线时已存在） → 确认非本次变更引入
    └── 测试环境问题 → 修复环境，重新运行
```

#### 5.3 回归证据记录

- 记录回归测试的完整输出
- 记录测试通过的数量和耗时
- 保存测试报告

---

### Step 6: 自我审查

**目标：在提交代码审查前，进行全面的自我检查**

#### 6.1 代码质量检查

- 代码风格是否与现有代码一致
- 是否有冗余代码或未使用的导入
- 是否有硬编码的值需要提取为配置
- 错误处理是否完善
- 日志记录是否充分

#### 6.2 变更完整性检查

- 是否所有变更需求都已实现
- 是否有遗漏的边界条件
- 是否有遗漏的错误处理
- 变更日志是否已更新

#### 6.3 测试完整性检查

- 回归测试是否覆盖了所有受影响功能
- 变更测试是否覆盖了正常/边界/异常路径
- 测试代码质量是否达标（命名清晰、断言充分）

#### 6.4 文档更新检查

- 是否需要更新技术文档
- 是否需要更新 API 文档
- 是否需要更新配置说明

---

### Step 7: 代码审查

**目标：通过两阶段审查确保变更质量**

#### 7.1 第一阶段：变更合规审查

审查变更是否符合客开场景的特殊要求：

- **变更范围审查**：是否超出了变更方案定义的范围
- **回归风险审查**：是否可能破坏已有功能
- **兼容性审查**：是否与现有架构兼容
- **向后兼容审查**：接口变更是否向后兼容

详细规范参见 `code-review-guide.md`。

#### 7.2 第二阶段：代码质量审查

审查代码质量（遵循 Paoding 代码审查标准）：

- 代码正确性
- 代码可读性
- 代码可维护性
- 测试充分性
- 性能影响

#### 7.3 审查记录

- 记录所有审查意见
- 对每个意见标注严重程度（阻塞/建议/讨论）
- 阻塞级别的问题必须修复后才能继续

---

### Step 8: 修复审查问题

**目标：修复代码审查中发现的所有问题**

#### 8.1 问题分类处理

| 严重程度 | 处理方式 | 验证要求 |
|----------|----------|----------|
| 阻塞（Blocker） | 必须修复 | 修复后重新运行相关测试 |
| 重要（Major） | 应该修复 | 修复后重新运行相关测试 |
| 建议（Minor） | 建议修复 | 记录处理决策 |
| 讨论（Discussion） | 讨论决定 | 记录讨论结果 |

#### 8.2 修复验证

- 修复后运行相关测试确认无破坏
- 如果修复引入了新的变更，需要补充变更测试
- 更新变更日志

---

### Step 9: 验证完成

**目标：进行最终的全面验证，确认变更可以交付**

#### 9.1 全量回归测试

- 运行完整的回归测试套件
- 确认所有测试通过
- 记录完整的测试报告

#### 9.2 变更验证

- 逐项验证每个变更需求是否已满足
- 验证变更的行为是否符合预期
- 验证变更的边界条件

#### 9.3 影响确认

- 确认变更的影响范围与方案一致
- 确认没有超出预期的副作用
- 确认性能无明显退化

#### 9.4 交付物整理

- 整理所有变更代码
- 整理所有测试代码
- 整理测试报告
- 整理代码审查记录
- 整理变更日志

---

## 铁律（不可违反）

### 铁律 1：没有回归测试保护就没有变更代码

> 在没有为受影响的已有功能编写回归测试之前，不允许修改任何已有代码。

**判断标准：**
- 所有受影响的已有功能都有对应的回归测试
- 所有回归测试在变更前都已通过（GREEN 基线）
- 回归测试覆盖了正常路径、边界条件和异常场景

### 铁律 2：没有根因调查就不能提出修复方案

> 当测试失败或出现 Bug 时，必须先调查根因，再提出修复方案。

**禁止行为：**
- 不看错误信息就修改代码
- 不理解失败原因就尝试修复
- "试试看"式的随机修改

### 铁律 3：完成声明必须有新鲜验证证据

> 声明任务完成时，必须附带最新的验证证据（包含回归测试通过证据）。

**要求：**
- 验证证据必须是最近一次运行的（不能使用旧的测试结果）
- 证据必须包含回归测试的完整输出
- 证据必须包含测试通过的数量和耗时

### 铁律 4：每段变更代码必须可追溯到变更需求

> 每段变更代码必须明确关联到对应的变更需求（CHG-ID）。

**要求：**
- 代码注释中标注变更ID
- Git 提交信息中关联变更ID
- 变更日志中记录变更ID

### 铁律 5：不理解的代码不改

> 在没有充分理解现有代码的逻辑之前，不允许修改该代码。

**判断标准：**
- 能够解释这段代码的作用
- 能够解释这段代码为什么这样写
- 能够预测修改后的影响

## 子代理协作

本阶段涉及多个子代理的协作：

### 开发者代理（Developer Agent）

**职责：** 执行变更实施的核心工作

- 编写回归测试和变更测试
- 实施代码变更
- 运行测试并分析结果
- 进行自我审查

**输入：** 实施计划、变更方案、架构理解文档
**输出：** 变更代码、测试代码、测试报告

### 审查代理（Reviewer Agent）

**职责：** 执行代码审查

- 变更合规审查（变更范围、回归风险、兼容性）
- 代码质量审查（正确性、可读性、可维护性）
- 生成审查意见和改进建议

**输入：** 变更代码、变更方案、代码审查指南
**输出：** 代码审查记录

### 调试代理（Debugger Agent）

**职责：** 当测试失败或出现 Bug 时进行系统化调试

- 分析错误信息和日志
- 定位问题根因
- 提出修复方案

**输入：** 错误信息、相关代码、系统日志
**输出：** 根因分析报告、修复建议

详细规范参见 `systematic-debugging-guide.md`。

### 验证代理（Verifier Agent）

**职责：** 执行最终的全面验证

- 运行全量回归测试
- 验证变更行为
- 确认影响范围
- 生成验证报告

**输入：** 变更代码、测试代码、回归测试计划
**输出：** 验证报告

## 出口条件（Gate 3 - 硬性门控）

以下所有条件必须全部满足，方可进入 Phase 4：

| # | 检查项 | 验证方式 |
|---|--------|----------|
| G3-1 | 所有变更需求已实现 | 变更需求-代码映射检查 |
| G3-2 | 所有回归测试通过（最新运行结果） | 测试报告检查 |
| G3-3 | 所有变更测试通过 | 测试报告检查 |
| G3-4 | 代码审查所有阻塞问题已修复 | 审查记录检查 |
| G3-5 | 每段变更代码可追溯到变更需求 | 代码-需求追溯检查 |
| G3-6 | 变更日志已更新 | 变更日志检查 |
| G3-7 | 无超出变更方案的意外修改 | 变更范围检查 |
| G3-8 | 完成声明附带新鲜验证证据 | 验证证据检查 |

**Gate 3 不通过的常见原因：**
- 回归测试失败（变更破坏了已有功能）
- 变更测试覆盖不完整（遗漏了边界条件）
- 代码审查有未修复的阻塞问题
- 变更超出了方案定义的范围
- 缺少新鲜的验证证据

## 红旗列表

遇到以下情况时，必须立即停止并报告：

| # | 红旗 | 说明 | 应对 |
|---|------|------|------|
| R3-1 | 回归测试基线无法建立 | 已有功能已经存在问题，无法建立GREEN基线 | 停止变更，先修复已有问题或与使用者协商 |
| R3-2 | 变更导致大量回归测试失败 | 变更的影响范围远超预期 | 停止变更，重新评估影响范围 |
| R3-3 | 发现变更方案与实际代码不一致 | 方案基于的假设与实际代码不符 | 停止变更，更新变更方案，重新评审 |
| R3-4 | 变更引入了安全漏洞 | 代码审查发现安全问题 | 立即停止，进行安全评估和修复 |
| R3-5 | 测试环境与生产环境不一致 | 测试结果可能不可靠 | 停止验证，先修复环境问题 |
| R3-6 | 变更依赖的第三方库有兼容性问题 | 新引入或升级的库与现有系统不兼容 | 评估替代方案或回退到兼容版本 |
| R3-7 | 发现未在方案中考虑的隐式依赖 | 变更影响了方案中未识别的模块 | 停止变更，更新影响分析，补充回归测试 |
| R3-8 | 数据迁移出现异常 | 数据变更导致数据不一致 | 立即停止，执行数据回滚，分析原因 |
