# 框架优化闭环指南

> **基于约束有效性分析，持续优化框架的约束体系。**
>
> 本指南定义了从分析到优化的完整闭环流程。

---

## 优化闭环总览

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐          │
│   │ 约束执行  │───→│ 数据收集  │───→│ 有效性   │          │
│   │          │    │          │    │ 分析     │          │
│   └──────────┘    └──────────┘    └────┬─────┘          │
│        ▲                               │                │
│        │                               ▼                │
│   ┌────┴─────┐    ┌──────────┐    ┌──────────┐          │
│   │ 效果验证  │←───│ 约束调整  │←───│ 优化建议  │          │
│   │          │    │          │    │          │          │
│   └──────────┘    └──────────┘    └──────────┘          │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## Step 1: 约束执行

在日常客开过程中，按照框架定义的约束执行各项工作。

### 执行要求

- 所有约束必须按定义执行
- 执行情况必须记录在约束追溯记录中
- 未执行的约束必须说明原因

### 执行记录

使用 [`constraint-trace-template.md`](./constraint-trace-template.md) 记录每次变更的约束执行情况。

---

## Step 2: 数据收集

在阶段和客开周期结束后，汇总约束执行数据。

### 收集内容

| 数据类型 | 来源 | 频率 |
|----------|------|------|
| 约束追溯记录 | 每次变更后填写 | 每次变更 |
| 迭代快照 | 阶段结束后汇总 | 每个阶段 |
| 聚合分析数据 | 客开周期结束后汇总 | 每个客开周期 |

### 数据质量要求

- **完整性**：所有变更都必须有对应的追溯记录
- **准确性**：追溯记录必须如实反映执行情况
- **及时性**：追溯记录必须在变更完成后及时填写

---

## Step 3: 有效性分析

基于收集的数据，分析约束的有效性。

### 分析维度

| 维度 | 分析内容 | 方法 |
|------|----------|------|
| 执行率分析 | 各约束的执行率是否达标 | 统计分析 |
| 趋势分析 | 约束执行率的变化趋势 | 时间序列分析 |
| 相关性分析 | 约束执行与客开质量的相关性 | 交叉分析 |
| 成本效益分析 | 约束的执行成本与收益 | 效率分析 |

### 分析输出

使用 [`aggregate-analysis-template.md`](./aggregate-analysis-template.md) 生成聚合分析报告。

---

## Step 4: 优化建议

基于分析结果，提出约束体系的优化建议。

### 建议类型

| 类型 | 触发条件 | 说明 |
|------|----------|------|
| **新增约束** | 发现了未被覆盖的风险场景 | 新增约束以覆盖新的风险点 |
| **调整约束** | 约束有效但执行成本过高 | 调整约束的严格程度或执行方式 |
| **淘汰约束** | 约束长期无效或成本远超收益 | 移除不再需要的约束 |
| **强化约束** | 约束有效但执行率不达标 | 加强约束的执行力度 |

### 建议评估标准

| 标准 | 说明 |
|------|------|
| **必要性** | 这个优化是否真的需要？ |
| **有效性** | 优化后预期能带来什么改善？ |
| **可行性** | 优化是否可以落地执行？ |
| **副作用** | 优化是否会引入新的问题？ |
| **优先级** | 与其他优化建议相比，优先级如何？ |

---

## Step 5: 约束调整

将评估通过的优化建议落实到框架中。

### 调整流程

```
优化建议 → 评估通过 → 编写调整方案 → 评审确认 → 落地实施 → 更新文档
```

### 调整范围

| 调整类型 | 涉及文件 | 审批要求 |
|----------|----------|----------|
| 新增铁律 | `core/iron-laws.md` | 必须用户确认 |
| 调整铁律 | `core/iron-laws.md` | 必须用户确认 |
| 新增反模式 | `core/anti-patterns.md` | AI 自主决定 |
| 调整反模式 | `core/anti-patterns.md` | AI 自主决定 |
| 新增客开约束 | `module-8-constraint-analytics/SKILL.md` | AI 自主决定 |
| 调整流程 | `core/workflow-overview.md` | 建议用户确认 |
| 调整模板 | 对应模板文件 | AI 自主决定 |

### 调整记录

每次约束调整必须记录以下信息：

| 项目 | 内容 |
|------|------|
| 调整编号 | ADJ-（自动编号） |
| 调整日期 | （YYYY-MM-DD） |
| 调整类型 | 新增 / 调整 / 淘汰 |
| 调整对象 | 具体的约束或流程 |
| 调整原因 | 基于什么分析结果 |
| 调整内容 | 具体的变更内容 |
| 预期效果 | 预期能带来什么改善 |
| 审批人 | AI 自主 / 用户确认 |

---

## Step 6: 效果验证

约束调整后，验证调整是否达到了预期效果。

### 验证方法

| 方法 | 说明 | 周期 |
|------|------|------|
| 执行率跟踪 | 跟踪调整后约束的执行率变化 | 每次变更 |
| 趋势对比 | 对比调整前后的趋势变化 | 每个阶段 |
| 效果评估 | 评估调整是否达到预期效果 | 每个客开周期 |

### 验证标准

| 结果 | 处理方式 |
|------|----------|
| 达到预期效果 | 确认调整，继续跟踪 |
| 部分达到效果 | 分析未达标原因，考虑进一步调整 |
| 未达到效果 | 回退调整，重新分析问题 |
| 产生副作用 | 评估副作用严重程度，决定是否回退 |

---

## 优化闭环的执行原则

1. **数据驱动**：所有优化决策必须基于数据，而非主观判断
2. **小步迭代**：每次只调整一个或少数几个约束，便于归因
3. **及时验证**：调整后尽快验证效果，不要等到下一个客开周期
4. **记录完整**：调整的全过程必须有完整记录
5. **保守原则**：对铁律的调整必须格外谨慎，宁可不调也不要调错

## 优化闭环与客开流程的关系

```
客开流程（Phase 0-5）
    │
    ├──→ 日常执行约束（Step 1）
    │         │
    │         ▼
    │    记录追溯数据（Step 2）
    │
    ├──→ 阶段结束
    │         │
    │         ▼
    │    生成迭代快照（Step 2）
    │
    └──→ 客开周期结束
              │
              ▼
         生成聚合分析报告（Step 3）
              │
              ▼
         提出优化建议（Step 4）
              │
              ▼
         实施约束调整（Step 5）
              │
              ▼
         效果验证（Step 6）
              │
              ▼
         进入下一个客开周期
```
