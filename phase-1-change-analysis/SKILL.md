# Phase 1: 变更需求分析（Change Analysis）

> **阶段定位**：不是从零收集需求，而是**分析变更需求 + 评估影响范围**，明确"要改什么、影响什么、风险在哪"。

---

## 概述

| 属性 | 说明 |
|------|------|
| **阶段名称** | 变更需求分析（Change Analysis） |
| **阶段编号** | Phase 1 |
| **核心目标** | 系统化地收集、分类、分析变更需求，评估影响范围和回归风险 |
| **阶段性质** | 分析与规划阶段，不产出任何代码变更 |

### 与从零开发的区别

| 维度 | 从零开发 | 客开（Customization） |
|------|----------|----------------------|
| 需求来源 | 用户原始需求 | 使用者提出的变更意向 |
| 分析重点 | 需求完整性和一致性 | 变更影响范围和回归风险 |
| 核心产出 | 需求规格说明书(SRS) | 变更规格文档(CRD) |
| 风险关注 | 需求遗漏 | 现有功能被破坏 |
| 评估维度 | 功能可行性 | 影响范围 + 回归风险 + AI能力匹配 |

---

## 入口条件

- [x] **Gate 0 已通过**：系统画像已完成且经过确认
- [x] **可行性评估已完成**：客开可行性等级为"高"或"中"
- [x] **系统画像文档可用**：作为影响分析的参考基础
- [x] **AI能力画像可用**：作为任务-AI匹配的参考基础
- [x] **使用者已明确变更意向**：至少有初步的变更方向

## 8步执行流程

### Step 1: 变更需求收集

**目标**：从使用者处收集所有变更需求，确保不遗漏。

**执行要点**：
1. **需求来源梳理**
   - 使用者口头/书面提出的变更要求
   - 系统使用过程中发现的问题和改进建议
   - 业务变化带来的系统调整需求
   - 技术升级或合规要求带来的变更
2. **需求获取方式**
   - 结构化访谈：按业务模块逐一确认变更需求
   - 系统走查：通过操作系统发现潜在变更点
   - 文档分析：从需求文档、用户反馈中提取变更项
3. **需求记录**
   - 每个变更需求记录为独立的变更请求（CR）
   - 使用 `change-request-template.md` 记录
   - 确保每个需求有明确的提出人和变更描述

**输出**：变更需求列表（初步）

---

### Step 2: 变更需求分类

**目标**：对收集到的变更需求进行分类，便于后续分析和优先级排序。

**执行要点**：
1. **按变更类型分类**

   | 类型 | 标识 | 说明 |
   |------|------|------|
   | 功能新增 | [新增] | 系统中不存在的新功能 |
   | 功能修改 | [修改] | 对已有功能的行为调整 |
   | 功能删除 | [删除] | 移除已有功能 |
   | 架构调整 | [架构] | 系统架构层面的变更 |
   | Bug修复 | [Bug] | 修复已知问题 |
   | 性能优化 | [性能] | 提升系统性能的变更 |
   | UI/UX调整 | [UI] | 界面和交互的调整 |
   | 数据调整 | [数据] | 数据模型或数据的变更 |

2. **按优先级分类**
   - P0 - 必须（阻塞业务）
   - P1 - 重要（严重影响体验）
   - P2 - 一般（改善性需求）
   - P3 - 可选（锦上添花）

3. **按复杂度分类**
   - 高复杂度：涉及多模块、多接口、数据模型变更
   - 中复杂度：涉及单模块、少量接口变更
   - 低复杂度：配置调整、UI微调、文案修改

**输出**：分类后的变更需求列表

---

### Step 3: 影响范围分析

**目标**：系统化地分析每个变更需求的影响范围。

**执行要点**：
1. **模块影响分析**
   - 变更直接影响哪些模块
   - 变更间接影响哪些模块（通过依赖关系）
   - 是否涉及不可动区域（参考系统画像第6章）
2. **接口影响分析**
   - 变更影响哪些 API 接口
   - 接口变更是否破坏向后兼容性
   - 是否影响外部系统对接
3. **数据影响分析**
   - 变更影响哪些数据表
   - 是否需要数据迁移
   - 是否影响数据一致性
4. **用户影响分析**
   - 变更影响哪些用户角色
   - 是否改变用户的操作习惯
   - 是否需要用户培训

**输出**：影响分析报告（`impact-analysis-template.md`）

---

### Step 4: 复用评估与争议点识别

**目标**：评估每个变更对现有模块/代码的复用情况，识别与现有架构的冲突点。

**执行要点**：

1. **现有资源复用评估**
   - 逐项评估每个变更需求涉及的现有模块、组件、工具类、配置
   - 对每个涉及的现有资源，标注复用判定：
     - ✅ **可直接复用**：现有模块/代码完全满足需求，无需修改
     - ⚠️ **需适配后复用**：现有模块/代码基本可用，但需要调整接口或逻辑
     - ❌ **不可复用**：现有模块/代码无法满足需求，需要重写或替换
     - 🆕 **需新增**：现有系统中不存在，需要从零开发
   - 记录复用评估结果到影响分析报告

2. **争议点识别**
   - 识别变更需求与现有架构/模块之间的冲突：
     - **架构冲突**：新功能要求与现有架构设计理念不一致
     - **模块冲突**：实现新功能需要修改现有模块的核心逻辑
     - **接口冲突**：新功能需要改变现有接口的签名或行为
     - **数据冲突**：新功能需要改变现有数据模型或数据格式
     - **先例缺失**：系统中没有类似功能的先例，实现方案存在不确定性
   - 对每个争议点，提供：
     - 争议描述：冲突的具体内容
     - 影响范围：如果按新功能要求修改，会影响哪些现有功能
     - 备选方案：至少提供2个解决方案（含各方案的优缺点和风险）
     - **决策请求**：标注需要使用者/决策者定夺的争议点

3. **级联影响链分析**
   - 对每个"需适配后复用"和"不可复用"的资源，追踪级联影响：
     - 修改模块A → 哪些模块调用了A → 这些模块是否需要同步修改 → 继续追踪...
     - 修改接口B → 哪些调用方依赖B → 调用方是否需要适配 → 继续追踪...
     - 修改数据模型C → 哪些代码依赖C的数据格式 → 是否需要数据迁移 → 继续追踪...
   - 绘制级联影响链路图
   - 评估级联修改的工作量和风险
   - 标注级联链中哪些修改是"必须的"，哪些是"建议的"

**输出**：复用评估报告 + 争议点清单 + 级联影响链（作为影响分析报告的一部分）

---

### Step 5: 变更依赖梳理

**目标**：分析变更之间的依赖关系，确定合理的执行顺序。

**执行要点**：
1. **变更间依赖分析**
   - 哪些变更必须在其他变更之前完成（前置依赖）
   - 哪些变更可以并行执行（无依赖）
   - 哪些变更之间存在冲突（互斥）
2. **执行顺序规划**
   - 基于依赖关系确定推荐的执行顺序
   - 标注关键路径（决定整体工期的变更序列）
   - 识别可以并行的工作流
3. **依赖风险识别**
   - 前置变更失败对后续变更的影响
   - 并行变更之间的潜在冲突

**输出**：变更依赖关系图 + 执行顺序建议

---

### Step 6: 回归风险识别

**目标**：识别每个变更可能引发的回归问题，确定回归测试范围。

**执行要点**：
1. **高风险变更点识别**
   - 涉及核心业务流程的变更
   - 涉及共享模块/公共组件的变更
   - 涉及数据模型变更的修改
   - 涉及权限/认证相关的变更
   - 涉及外部系统对接的变更
2. **回归测试范围确定**
   - **必须回归**：与变更直接相关的功能，必须100%回归
   - **建议回归**：与变更间接相关的功能，建议抽样回归
   - **可选回归**：与变更关联度低的功能，可选回归
3. **回归测试策略**
   - 手动测试 vs 自动化测试
   - 测试环境要求
   - 测试数据准备

**输出**：回归测试范围清单（作为影响分析报告的一部分）

---

### Step 7: 变更需求对齐确认

**目标**：向使用者展示完整的分析结果，确认变更范围和优先级。

**执行要点**：
1. **分析结果展示**
   - 逐项展示每个变更的影响分析
   - 重点说明高风险变更和回归风险
   - 展示变更间的依赖关系和执行顺序
2. **范围确认**
   - 确认变更范围的完整性（是否有遗漏）
   - 确认变更范围的合理性（是否有不必要的变更）
   - 确认优先级排序
3. **风险确认**
   - 确认使用者对风险的理解和接受度
   - 确认风险缓解措施
4. **使用 `change-alignment-checklist.md` 逐项检查**

**输出**：变更对齐确认记录

---

### Step 8: 变更规格文档输出

**目标**：整合所有分析结果，输出正式的变更规格文档。

**执行要点**：
1. **整合变更请求**
   - 将所有确认的变更请求整合为统一的变更规格文档（CRD）
   - 按执行顺序组织文档结构
2. **补充技术细节**
   - 为每个变更补充技术实现要点
   - 明确验收标准
3. **文档审核**
   - 内部审核：确保技术分析的准确性
   - 使用者审核：确保业务理解的正确性
4. **文档定稿**
   - 确认所有变更请求的优先级和范围
   - 确认影响分析和回归测试范围
   - 获得双方签字确认

**输出**：变更规格文档（CRD）

## 6条铁律

### 铁律一：变更需求必须有影响分析
> 每一个变更需求都必须经过影响范围分析。没有影响分析的变更请求不得进入执行阶段。影响分析是客开安全的第一道防线。

### 铁律二：不评估影响不改代码
> 在影响分析完成并经过确认之前，不对任何代码进行修改。理解影响范围是修改代码的必要前提。

### 铁律三：变更范围必须确认
> 变更需求的范围、优先级、影响分析结果必须经过使用者确认。未确认的变更不得执行。确认是责任的分界线。

### 铁律四：回归风险不可忽视
> 客开最大的风险不是新功能做不出来，而是改坏了已有功能。每一个变更都必须评估回归风险，确定回归测试范围。

### 铁律四-B：争议点必须暴露，不可隐瞒
> 当变更需求与现有架构/模块存在冲突时，必须将争议点完整暴露给决策者，由决策者定夺。AI不得自行决定有争议的方案，也不得隐瞒争议点。

### 铁律四-C：级联影响必须追踪到底
> 当变更涉及修改现有模块/接口/数据时，必须追踪完整的级联影响链。未追踪完级联影响的变更不得进入执行阶段。

### 铁律五：依赖关系必须理清
> 变更之间的依赖关系必须明确。错误的执行顺序会导致返工和冲突。依赖分析是执行计划的基础。

### 铁律六：不可动区域不可触碰
> 系统画像中标识的不可动区域，在任何变更中都不允许触碰。如果变更需求与不可动区域冲突，必须重新评估方案。

## 红旗列表

以下情况出现时，必须立即暂停并升级处理：

| 序号 | 红旗 | 说明 | 处理方式 |
|------|------|------|----------|
| R1-1 | 变更涉及不可动区域 | 变更需求与系统画像中的不可动区域冲突 | 暂停，重新评估方案 |
| R1-2 | 影响范围无法确定 | 无法评估变更的完整影响范围 | 暂停，深入分析或寻求人工帮助 |
| R1-3 | 变更间存在不可调和的冲突 | 两个变更需求互相矛盾 | 暂停，与使用者协商优先级 |
| R1-4 | 回归风险过高 | 变更可能破坏核心业务且无法有效回归测试 | 暂停，评估是否需要先建立测试基线 |
| R1-5 | 使用者频繁变更需求 | 变更范围不稳定，频繁增删改 | 暂停，重新对齐需求范围 |
| R1-6 | 变更需求超出AI能力范围 | 存在AI无法胜任的变更任务 | 暂停，评估人工介入方案 |
| R1-7 | 缺少关键信息 | 无法获取影响分析所需的关键信息 | 暂停，协调信息获取 |
| R1-8 | 数据变更无回滚方案 | 涉及不可逆的数据变更且无回滚方案 | 暂停，必须先制定数据回滚方案 |
