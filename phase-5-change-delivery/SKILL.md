# Phase 5: 变更交付（Change Delivery）

> **定位：客开的交付不是交付整个系统，而是交付"变更集"——核心是"变更正确 + 无回归"**

---

## 概述

本阶段是客开流程的最终交付阶段。与从零开发不同，客开的交付对象不是整个系统，而是一个**变更集**（Change Set）——一组经过验证的、对现有系统的修改。

客开交付的核心标准不是"功能是否完整"，而是：

1. **变更正确**：所有计划的变更都已正确实施
2. **无回归**：变更没有破坏任何已有功能
3. **可追溯**：每段变更代码都可以追溯到变更需求
4. **可回滚**：如果出现问题，可以安全回滚
5. **有文档**：变更相关的文档都已更新

本阶段采用严格的"检查-验证-确认-打包"流程，确保交付质量。

---

## 入口条件

| 条件 | 说明 |
|------|------|
| Gate 4 通过（或知识沉淀基本完成） | Phase 4 的软性门控已通过，核心检查项满足 |
| 变更代码已全部提交 | 所有变更代码已通过审查并提交到代码仓库 |
| 回归测试全部通过 | 全量回归测试通过，无破坏 |
| 知识沉淀已完成 | 变更回顾、错题本更新、系统知识库更新已完成 |

---

## 6步执行流程

### Step 1: 变更完整性检查

**目标：确认所有计划的变更都已正确完成，无遗漏**

#### 1.1 变更需求覆盖检查

- 逐项检查变更规格文档（CRD）中的每个变更需求
- 确认每个变更需求都有对应的代码实现
- 确认每个变更需求的验收标准都已满足

#### 1.2 变更代码完整性检查

- 确认所有计划的代码变更都已提交
- 确认没有超出计划的意外修改
- 确认变更代码的注释和提交信息完整

#### 1.3 变更清单核对

```
变更规格文档中的变更清单
├── CHG-001 → 代码已实现 → 测试已通过 → [通过]
├── CHG-002 → 代码已实现 → 测试已通过 → [通过]
├── CHG-003 → 代码已实现 → 测试已通过 → [通过]
└── ...
```

**输出：** 变更完整性检查报告

---

### Step 2: 全量回归测试

**目标：运行完整的回归测试套件，确认变更没有破坏任何已有功能**

#### 2.1 回归测试执行

- 运行完整的回归测试套件（不跳过任何测试）
- 记录完整的测试输出
- 确认所有测试通过

#### 2.2 测试结果分析

```
全量回归测试结果
├── 总测试数：[数量]
├── 通过数：[数量]
├── 失败数：0
├── 跳过数：0（不允许跳过）
├── 执行耗时：[时间]
└── 覆盖率：[百分比]
```

#### 2.3 性能回归检查

- 如果变更涉及性能相关模块，运行性能基准测试
- 对比变更前后的性能指标
- 确认无性能退化

**输出：** 回归测试报告

---

### Step 3: 变更追溯验证

**目标：建立"变更代码 → 变更需求 → 影响分析"的完整追溯链**

#### 3.1 代码到需求的追溯

- 每段变更代码都能追溯到对应的变更需求（CHG-ID）
- 每个变更需求都有对应的代码实现
- 不存在"孤儿变更"（无法追溯到需求的代码变更）

#### 3.2 需求到影响的追溯

- 每个变更需求都有对应的影响分析
- 影响分析中标注的所有受影响模块都已验证
- 影响分析中的回归测试范围都已覆盖

#### 3.3 追溯矩阵

| 变更编号 | 变更描述 | 涉及文件 | 影响模块 | 回归测试 | 状态 |
|----------|----------|----------|----------|----------|------|
| CHG-001 | [描述] | [文件列表] | [模块列表] | [测试列表] | [已追溯] |
| CHG-002 | [描述] | [文件列表] | [模块列表] | [测试列表] | [已追溯] |

**输出：** 追溯验证报告

---

### Step 4: 文档完整性检查

**目标：确认所有变更相关的文档都已更新**

#### 4.1 变更文档检查

- 变更规格文档（CRD）是否已更新为最终版本
- 变更方案文档是否已更新为实际实施方案
- 影响分析文档是否已更新为实际影响

#### 4.2 系统知识库检查

- 系统知识库是否已更新（Phase 4 的产出）
- 新发现的隐式依赖是否已记录
- 地雷区清单是否已更新

#### 4.3 部署文档检查

- 部署说明是否已编写
- 环境配置变更是否已记录
- 数据库变更脚本是否已准备

#### 4.4 回滚文档检查

- 回滚方案是否已编写
- 回滚步骤是否可执行
- 回滚验证方法是否明确

**输出：** 文档完整性检查报告

---

### Step 5: 使用者验收

**目标：让使用者确认变更满足需求，原有功能未受影响**

#### 5.1 功能演示

- 向使用者演示所有变更的功能
- 逐项确认每个变更需求是否满足
- 记录使用者的反馈

#### 5.2 回归确认

- 向使用者确认原有功能未受影响
- 让使用者操作关键业务流程，验证无异常
- 记录使用者的回归确认

#### 5.3 遗留问题确认

- 确认是否有未解决的遗留问题
- 确认遗留问题的处理计划
- 获得使用者对遗留问题的确认

#### 5.4 最终签字

- 使用者签署交付确认记录
- 确认变更可以进入部署阶段

**输出：** 交付确认记录

---

### Step 6: 交付打包

**目标：将所有交付物打包为完整的变更交付包**

#### 6.1 交付物清单

使用 `change-deliverables-template.md` 整理所有交付物：

- 变更代码（含 Git 提交记录）
- 测试代码（回归测试 + 变更测试）
- 测试报告
- 变更文档
- 系统知识库更新
- 部署说明
- 回滚方案
- 追溯验证报告
- 交付确认记录

#### 6.2 交付包结构

```
delivery-package/
├── 01-change-code/          # 变更代码
│   ├── git-diff.patch       # 代码差异
│   └── git-log.txt          # 提交记录
├── 02-test-code/            # 测试代码
│   ├── regression/          # 回归测试
│   └── change/              # 变更测试
├── 03-test-reports/         # 测试报告
│   ├── regression-report.html
│   └── change-report.html
├── 04-documents/            # 文档
│   ├── change-spec.md       # 变更规格文档
│   ├── impact-analysis.md   # 影响分析
│   └── knowledge-update.md  # 知识库更新
├── 05-deployment/           # 部署
│   ├── deploy-guide.md      # 部署说明
│   └── rollback-guide.md    # 回滚方案
├── 06-verification/         # 验证
│   ├── traceability.md      # 追溯验证报告
│   └── acceptance.md        # 验收确认记录
└── README.md                # 交付包说明
```

#### 6.3 交付包校验

- 确认交付包中所有文件完整
- 确认交付包中的文档与实际代码一致
- 确认交付包可以在干净环境中部署

**输出：** 变更交付包

## 4条铁律

### 铁律 1：不完整不交付

> 变更清单上的每一项都必须完成。不允许"部分交付"或"带病交付"。

**判断标准：**
- 变更规格文档中的每个变更需求都有对应的代码实现
- 每个变更需求的验收标准都已满足
- 不存在未完成的变更项

### 铁律 2：未回归验证不交付

> 全量回归测试必须通过，不允许跳过任何测试。回归测试是客开交付的底线。

**判断标准：**
- 全量回归测试通过率 100%
- 不存在跳过或忽略的测试
- 测试报告包含新鲜的验证证据

### 铁律 3：无文档不交付

> 变更相关的文档必须更新。没有文档的变更交付是不可接受的。

**判断标准：**
- 变更文档已更新为最终版本
- 部署说明已编写
- 回滚方案已编写
- 系统知识库已更新

### 铁律 4：使用者未确认不交付

> 使用者必须确认变更满足需求且原有功能未受影响。未经使用者确认的变更不得交付。

**判断标准：**
- 使用者已确认所有变更需求满足
- 使用者已确认原有功能未受影响
- 使用者已签署交付确认记录

## 出口条件（Gate 5 - 硬性门控）

> Gate 5 为硬性门控：所有条件必须全部满足，方可完成交付。

| # | 检查项 | 验证方式 | 状态 |
|---|--------|----------|------|
| G5-1 | 所有变更需求已完成 | 变更需求-代码映射检查 | [ ] |
| G5-2 | 全量回归测试100%通过 | 测试报告检查 | [ ] |
| G5-3 | 无跳过或忽略的测试 | 测试报告检查 | [ ] |
| G5-4 | 每段变更代码可追溯到需求 | 追溯矩阵检查 | [ ] |
| G5-5 | 无"孤儿变更" | 代码差异检查 | [ ] |
| G5-6 | 变更文档已更新 | 文档完整性检查 | [ ] |
| G5-7 | 部署说明已编写 | 部署文档检查 | [ ] |
| G5-8 | 回滚方案已编写 | 回滚文档检查 | [ ] |
| G5-9 | 使用者已确认变更满足需求 | 验收确认记录检查 | [ ] |
| G5-10 | 使用者已确认无回归问题 | 回归确认记录检查 | [ ] |
| G5-11 | 交付包完整且可部署 | 交付包校验 | [ ] |
| G5-12 | 交付确认记录已签署 | 签署记录检查 | [ ] |

**Gate 5 不通过的常见原因：**
- 存在未完成的变更需求
- 回归测试存在失败或跳过的用例
- 存在无法追溯到需求的"孤儿变更"
- 部署说明或回滚方案缺失
- 使用者未完成验收确认

## 红旗列表

遇到以下情况时，必须立即停止并报告：

| # | 红旗 | 说明 | 应对 |
|---|------|------|------|
| R5-1 | 回归测试发现新的失败 | 全量回归测试中发现了之前未发现的失败 | 停止交付，分析失败原因，修复后重新测试 |
| R5-2 | 发现未在方案中的代码变更 | 交付检查发现存在超出计划的代码修改 | 停止交付，分析变更原因，评估影响 |
| R5-3 | 使用者发现功能不符合预期 | 验收过程中使用者发现变更行为与预期不符 | 停止交付，与使用者对齐需求，必要时回退到 Phase 3 |
| R5-4 | 部署方案不可行 | 部署说明中的步骤无法在目标环境执行 | 停止交付，重新设计部署方案 |
| R5-5 | 回滚方案不可靠 | 回滚方案无法保证安全回滚 | 停止交付，重新设计回滚方案 |
| R5-6 | 交付包不完整 | 交付包中缺少必要的文件或信息 | 停止交付，补全缺失的交付物 |
| R5-7 | 性能回归超过阈值 | 性能基准测试显示性能退化超过可接受范围 | 停止交付，分析性能退化原因，优化后重新测试 |
| R5-8 | 安全审查发现问题 | 交付前的安全检查发现潜在安全风险 | 停止交付，进行安全评估和修复 |
