# Phase 2: 架构理解与变更方案

> **定位：不是从零设计架构，而是深入理解现有架构 → 设计变更方案 → 评估兼容性**

---

## 概述

本阶段是客开（定制化开发）流程中最关键的阶段之一。客开场景与从零开发不同——我们面对的是一个已经运行、已经承载业务、已经积累了大量隐式知识的系统。任何变更都必须建立在对现有架构的深入理解之上。

本阶段的核心目标是：

1. **深入理解现有系统架构**——不是表面浏览，而是真正理解"为什么这样设计"
2. **设计变更方案**——针对每个变更需求，设计最小化影响的技术方案
3. **评估兼容性**——确保变更不会破坏现有功能
4. **制定可执行的实施计划**——无占位符、每步可验证

---

## 入口条件

| 条件 | 说明 |
|------|------|
| Gate 1 通过 | Phase 1（需求理解与变更规格）的硬性门控已通过 |
| 变更规格文档已完成 | 所有变更需求已明确、已确认、无歧义 |

| 必要的上下文信息 | 已获得系统文档、架构说明、部署信息等 |

## 执行流程

### Step 1: 现有架构深入分析

**目标：建立对现有系统的全面、深入理解**

#### 1.1 代码阅读与模块识别

- 从入口点开始追踪代码执行路径
- 识别所有核心模块及其职责边界
- 绘制模块依赖关系图（包括编译依赖和运行时依赖）
- 识别各模块的公开接口与内部实现

#### 1.2 设计模式识别

- 识别系统中使用的设计模式（工厂、策略、观察者、装饰器等）
- 理解每个设计模式的使用动机和适用场景
- 识别反模式或技术债务

#### 1.3 隐式依赖发现

- 搜索硬编码的配置值、魔法数字
- 识别隐式的状态依赖（如全局变量、单例状态）
- 发现数据库层面的隐式约束（触发器、存储过程、外键级联）
- 识别跨模块的数据格式约定

#### 1.4 历史演进理解

- 通过 Git 历史理解代码的演进脉络
- 识别"为什么这样写"的历史原因
- 发现被废弃但未清理的代码路径

**输出：** 架构理解文档（使用 `architecture-understanding-template.md` 模板）

---

### Step 2: 变更方案设计

**目标：针对每个变更需求设计具体的技术方案**

#### 2.1 变更需求映射

将每个变更需求映射到具体的代码位置和模块：

```
变更需求 → 影响模块 → 影响文件 → 影响函数/类 → 变更类型（新增/修改/删除）
```

#### 2.2 方案设计原则

- **最小化变更**：用最少的代码改动实现需求
- **保持一致性**：新代码风格与现有代码保持一致
- **利用现有机制**：优先使用系统已有的扩展点、钩子、配置机制
- **避免引入新依赖**：除非绝对必要，不引入新的第三方库

#### 2.3 方案设计内容

对每个变更需求，输出：

- **当前实现分析**：现有代码如何工作
- **变更方案描述**：具体如何修改，包含伪代码或代码片段
- **影响范围标注**：哪些模块、接口、数据结构会受到影响
- **兼容性分析**：与现有架构的兼容性评估
- **替代方案**：至少考虑一个备选方案并说明取舍
- **复用评估**：标注涉及的现有模块/代码的复用判定（✅可复用 / ⚠️需适配 / ❌不可复用 / 🆕需新增）
- **争议点处理**：如果该变更涉及 Phase 1 识别的争议点，说明采用的解决方案及理由
- **级联影响说明**：如果该变更会引发级联修改，说明级联链路和各层级的修改内容

**输出：** 变更方案文档（使用 `change-plan-template.md` 模板）

---

### Step 3: 兼容性评估

**目标：确保变更方案与现有系统完全兼容**

#### 3.1 架构兼容性

- 新方案是否符合现有架构的设计理念
- 是否违反现有的分层原则、依赖方向
- 是否引入了架构层面的不一致

#### 3.2 接口兼容性

- 修改的接口是否影响其他调用方
- 接口签名变更是否向后兼容
- 数据格式变更是否需要数据迁移

#### 3.3 数据兼容性

- 数据库 Schema 变更是否影响现有数据
- 是否需要数据迁移脚本
- 缓存数据是否需要失效处理

#### 3.4 行为兼容性

- 变更是否改变了现有功能的行为
- 是否有隐式的行为契约被破坏
- 并发场景下是否有新的竞态条件

---

### Step 3.5: 争议点决策与级联影响确认

**目标：对 Phase 1 识别的争议点做出技术方案建议，确认级联影响的完整性和处理方案**

#### 3.5.1 争议点技术方案细化

对 Phase 1 识别的每个争议点（DIS-XXX），提供详细的技术方案：

- **方案A（推荐）**：
  - 具体实现方式（含代码层面的描述）
  - 对现有功能的影响（哪些功能会受影响、影响程度）
  - 级联修改清单（需要同步修改的模块/接口/数据）
  - 工作量估算
  - 风险评估
  - 回滚方案

- **方案B（备选）**：
  - 同上

- **对比分析**：
  - 两个方案在功能完整性、稳定性、工作量、风险方面的对比
  - 推荐理由

- **决策请求**：
  - 明确标注需要决策者定夺的内容
  - 提供决策所需的关键信息（非技术人员能理解的风险说明）

#### 3.5.2 级联影响详细分析

对每个会引发级联修改的变更，输出详细的级联影响分析：

- **级联链路图**：完整的修改传播路径
- **各层级修改清单**：
  - L1（直接影响）：必须修改的模块/接口/数据
  - L2（间接影响）：建议修改的模块/接口/数据
  - L3+（可能影响）：需要评估的模块/接口/数据
- **级联修改工作量汇总**：总工作量估算
- **级联风险点**：级联链中风险最高的环节
- **级联回归范围**：级联修改需要额外回归测试的功能清单

#### 3.5.3 角色化方案输出

根据 Phase 0 识别的利益相关者角色，生成不同详细程度的方案报告：

- **完整技术版**（给技术评审者）：
  - 全部技术细节：代码变更、架构影响、兼容性分析、级联影响链、技术风险评估
  - 包含所有备选方案的技术对比
  - 包含详细的回滚方案

- **业务摘要版**（给决策者、业务验证者）：
  - 功能影响说明：哪些功能会受影响、影响程度
  - 风险点清单：用非技术语言描述的风险（如"修改用户模块可能导致登录功能暂时不可用"）
  - 争议点及决策选项：每个争议点的备选方案及其对业务的影响
  - 工期估算和资源需求
  - 不包含：代码细节、架构实现方案

- **简要说明版**（给使用者）：
  - 变更内容概述：做了什么改动
  - 操作变化说明：使用方式有什么变化
  - 功能影响说明：哪些功能会受影响
  - 不包含：技术细节、风险评估细节、争议点

**输出：** 争议点决策文档 + 级联影响详细分析 + 角色化方案报告（三个版本）

---

### Step 4: 变更实施计划

**目标：制定详细的、可执行的实施计划**

#### 4.1 任务拆解

将变更方案拆解为具体的实施任务：

- 每个任务必须明确：涉及哪些文件、具体做什么操作
- 每个任务必须可独立验证（有明确的验证标准）
- 任务粒度：一个任务应在 1-4 小时内可完成

#### 4.2 依赖关系梳理

- 明确任务之间的前置/后置关系
- 识别可以并行的任务
- 绘制任务依赖关系图

#### 4.3 执行顺序规划

- 按照依赖关系确定执行顺序
- 高风险变更优先实施（留出更多缓冲时间）
- 基础设施变更（如数据库变更）优先于业务逻辑变更

#### 4.4 回归测试计划

- 基于影响范围确定回归测试范围
- 为每个任务指定对应的回归测试命令
- 确定测试优先级

**输出：** 实施计划（包含在 `change-plan-template.md` 中）

---

### Step 5: 回归测试计划

**目标：基于 Phase 1 的回归范围，设计具体的测试策略**

#### 5.1 测试范围确定

- 基于 Phase 1 识别的功能影响范围
- 基于架构分析发现的隐式依赖
- 基于变更方案的影响范围标注

#### 5.2 测试策略设计

- **基线测试**：变更前必须通过的现有功能测试
- **变更测试**：验证变更行为是否符合预期
- **集成测试**：验证变更后模块间交互是否正常
- **边界测试**：针对变更引入的边界条件

#### 5.3 测试优先级

| 优先级 | 说明 | 示例 |
|--------|------|------|
| P0 - 核心 | 变更直接影响的核心功能 | 订单创建流程（如果变更了订单模块） |
| P1 - 重要 | 与变更模块有直接交互的功能 | 支付模块（如果订单模块变更影响了支付接口） |
| P2 - 一般 | 与变更模块有间接交互的功能 | 报表模块（如果订单数据结构变更） |
| P3 - 低 | 受影响概率较低的功能 | 系统设置页面 |

**输出：** 回归测试计划（包含在 `change-plan-template.md` 中）

---

### Step 6: 多轮专家验证与 QA 独立门控

**目标：通过分领域专家独立验证 + QA 独立质量门控，确保方案质量**

> 参考 AIOX 框架的多代理验证机制，将方案评审从"一轮通用评审"升级为"多轮分领域专家验证 + QA 独立门控"。

#### 6.1 第一轮：领域专家验证（并行）

各领域专家独立审查方案中与自己领域相关的部分，互不干扰：

| 专家角色 | 审查范围 | 审查重点 | 输出 |
|----------|----------|----------|------|
| **架构专家** | 架构理解文档 + 变更方案 | 架构兼容性、设计合理性、模块边界 | `review-architecture.md` |
| **数据专家** | 数据模型变更 + 数据兼容性 | Schema 变更、数据迁移、数据一致性 | `review-database.md` |
| **安全专家** | 安全相关变更 + 认证授权 | 安全风险、权限变更、数据安全 | `review-security.md` |
| **业务专家** | 业务逻辑变更 + 用户体验 | 业务规则正确性、用户体验影响 | `review-business.md` |

每个专家的审查流程：
1. 阅读方案中与自己领域相关的部分
2. 逐项检查（使用 `design-review-checklist.md` 对应维度）
3. **验证**：确认发现的问题是否真实存在（非误判）
4. **估算**：对每个问题估算修复工作量和优先级
5. **补充**：识别方案中遗漏的问题
6. 输出审查报告，结论为：通过 / 需修改 / 阻塞

#### 6.2 第二轮：方案修订

基于第一轮各专家的审查意见，修订方案：
1. 汇总所有专家的 FAIL 项和 WARN 项
2. 按优先级排序（阻塞 > 高 > 中 > 低）
3. 逐项修订方案
4. 对每个修订项记录：问题 → 修订内容 → 修订理由

#### 6.3 第三轮：QA 独立质量门控

QA 代理独立执行最终质量门控，不参与前两轮：

| 检查维度 | 检查内容 | 通过标准 |
|----------|----------|----------|
| **完整性检查** | 所有专家的 FAIL 项是否已全部修复 | 0 个未修复的 FAIL 项 |
| **一致性检查** | 修订后的方案各部分是否一致 | 无自相矛盾 |
| **交叉影响检查** | 修复某个问题是否引入了新问题 | 无新增的高风险项 |
| **可执行性检查** | 实施计划是否可直接执行 | 无占位符、每步可验证 |
| **回归覆盖检查** | 回归测试范围是否覆盖所有受影响功能 | 与影响分析一致 |
| **争议点检查** | 所有争议点是否已有决策方案 | 0 个未决策的争议点 |

**QA 门控决策**：
- ✅ **APPROVED**：所有维度通过 → 进入 Step 7
- ❌ **NEEDS WORK**：存在未通过项 → 回退到 Step 6.2 重新修订（最多回退 2 次）

#### 6.4 评审记录存档

所有评审记录存档到 `01-docs/phase-2/reviews/`：
- `review-architecture.md` — 架构专家审查报告
- `review-database.md` — 数据专家审查报告
- `review-security.md` — 安全专家审查报告
- `review-business.md` — 业务专家审查报告
- `review-qa-gate.md` — QA 质量门控报告
- `review-summary.md` — 评审汇总（含修订记录）

**输出：** 多轮评审记录 + QA 门控报告

---

### Step 7: 方案确认与进入下一阶段

**目标：获得所有相关方的确认，进入实施阶段**

#### 7.1 方案确认

- 向使用者（需求方）展示变更方案，确认技术方案满足业务需求
- 向技术负责人展示架构评估结果，确认方案的技术可行性
- 获得明确的确认记录

#### 7.2 进入条件确认

- 确认所有必输出物已完成
- 确认 Gate 2 所有检查项已通过
- 确认实施所需的资源和权限已就绪

#### 7.3 可执行任务生成

> 参考 AIOX 框架的 Epic/Story 生成机制，将变更方案转化为可直接执行的开发任务。

将变更方案中的实施计划转化为标准化的可执行任务：

| 任务ID | 任务描述 | 涉及文件 | 操作步骤 | 验证命令 | 关联变更 | 前置任务 | 预估工时 |
|--------|----------|----------|----------|----------|----------|----------|----------|
| TASK-001 | | | | | CHG-001 | 无 | 2h |
| TASK-002 | | | | | CHG-001 | TASK-001 | 1h |

每个任务必须满足：
- **可独立执行**：有明确的操作步骤，不需要额外上下文
- **可独立验证**：有可执行的验证命令，不需要人工判断
- **粒度合适**：1-4 小时可完成
- **依赖明确**：前置任务已标注

**输出：** 可执行任务清单（包含在变更方案文档中）

## 铁律（不可违反）

### 铁律 1：不理解现有架构不改代码

> 在没有充分理解现有架构之前，不允许设计任何变更方案。

**判断标准：** 能够回答以下问题才算"理解"：
- 这个模块为什么这样设计？
- 修改这里会影响哪些其他模块？
- 有哪些隐式的依赖关系？

### 铁律 2：变更方案必须评估兼容性

> 每个变更方案必须包含兼容性评估，未评估兼容性的方案视为不完整。

**评估维度：** 架构兼容性、接口兼容性、数据兼容性、行为兼容性。

### 铁律 3：计划不允许占位符

> 实施计划中不允许出现"待定""TODO""视情况而定"等占位符。

**要求：** 每个任务必须有明确的操作描述、涉及文件、验证标准。

### 铁律 4：每个变更任务必须可验证

> 每个变更任务必须附带可执行的验证命令或验证步骤。

**要求：** 验证命令必须能够直接执行，不能依赖人工判断。

### 铁律 5：变更方案必须标注影响范围

> 每个变更必须明确标注其影响范围，包括直接影响和间接影响。

**要求：** 影响范围必须具体到模块、文件、函数级别。

### 铁律 5-B：争议点必须有备选方案

> 当变更方案涉及争议点时，必须提供至少2个备选方案，并标注推荐方案及理由。不得只提供一个方案让决策者"同意或不同意"。

### 铁律 5-C：级联影响必须量化

> 级联影响不能只说"可能影响其他模块"，必须具体到：影响哪些模块、每个模块需要什么修改、工作量多少、风险等级多高。模糊的级联评估等于没有评估。

### 铁律 6：方案变更必须追溯

> 任何对已确认方案的修改，必须记录变更原因、变更内容、影响分析。

**要求：** 使用变更日志记录所有方案修改，确保可追溯。

## 出口条件（Gate 2 - 硬性门控）

以下所有条件必须全部满足，方可进入 Phase 3：

| # | 检查项 | 验证方式 |
|---|--------|----------|
| G2-1 | 架构理解文档已完成，覆盖所有受影响模块 | 文档完整性检查 |
| G2-2 | 每个变更需求都有对应的技术方案 | 方案-需求映射检查 |
| G2-3 | 所有变更方案都包含兼容性评估 | 兼容性评估检查 |
| G2-3.5 | 争议点已有决策方案 | 所有争议点都有至少2个备选方案，决策请求已提交 | [ ] |
| G2-3.6 | 级联影响已量化 | 所有级联影响已具体到模块/接口级别，工作量已估算 | [ ] |
| G2-3.7 | 角色化报告已生成 | 完整技术版、业务摘要版、简要说明版均已生成 | [ ] |
| G2-4 | 实施计划无占位符，每个任务可验证 | 计划逐项检查 |
| G2-5 | 回归测试计划覆盖所有受影响功能 | 测试覆盖度检查 |
| G2-6 | 方案评审所有检查项通过 | 评审记录检查 |
| G2-6.5 | QA 独立门控通过 | QA 质量门控结论为 APPROVED | [ ] |
| G2-6.6 | 可执行任务已生成 | 所有变更已转化为可执行任务，每任务可独立执行和验证 | [ ] |
| G2-7 | 使用者已确认变更方案 | 确认记录检查 |
| G2-8 | 所有高风险变更都有回滚方案 | 回滚方案检查 |

**Gate 2 不通过的常见原因：**
- 架构理解停留在表面，未深入到代码层面
- 变更方案缺少兼容性分析
- 实施计划中有"待定"类占位符
- 回归测试范围未覆盖间接影响的功能
- 高风险变更缺少回滚方案

## 红旗列表

遇到以下情况时，必须立即停止并报告：

| # | 红旗 | 说明 | 应对 |
|---|------|------|------|
| R2-1 | 发现未文档化的核心业务逻辑 | 现有系统存在关键逻辑没有文档说明 | 停止方案设计，先补充文档 |
| R2-2 | 发现架构层面的严重技术债务 | 如循环依赖、全局状态滥用 | 评估是否需要在变更前先修复技术债务 |
| R2-3 | 变更需求与现有架构根本冲突 | 需求要求的功能与架构设计理念不兼容 | 立即与使用者沟通，讨论替代方案 |
| R2-4 | 影响范围超出预期 | 变更影响范围远大于初始评估 | 重新评估影响范围，更新回归测试计划 |
| R2-5 | 发现安全或合规风险 | 变更可能引入安全漏洞或违反合规要求 | 立即停止，进行安全评估 |
| R2-6 | 依赖的第三方库存在已知漏洞 | 变更需要引入或升级有漏洞的依赖 | 评估替代方案或先修复漏洞 |
| R2-7 | 数据迁移方案不可逆 | 数据变更无法安全回滚 | 设计可逆的迁移方案或放弃该变更 |
| R2-8 | 现有系统缺乏足够的测试覆盖 | 回归测试无法有效保护已有功能 | 在变更前先补充关键测试 |
