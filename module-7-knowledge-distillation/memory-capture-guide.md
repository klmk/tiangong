# 记忆捕获指南

> **如何识别和记录有价值的知识碎片。**
>
> 本指南定义了知识碎片的识别信号、记录方法和优先级判断。

---

## 什么是知识碎片

知识碎片是在客开过程中产生的、有价值的但尚未结构化的信息片段。它们是知识蒸馏的原始素材。

```
知识碎片（原始素材）
    │
    ▼ 蒸馏
结构化认知（project-brain.md）
    │
    ▼ 应用
优化后的行为
```

---

## 记忆捕获的触发信号

### 通用触发信号（与 Paoding 相同）

| 触发信号 | 知识类型 | 示例 |
|----------|----------|------|
| "原来这个模块是这样工作的" | 系统理解 | 理解了某个模块的内部机制 |
| "这个设计模式在这里用得很好" | 编码认知 | 发现了好的设计实践 |
| "这里有个坑，要注意" | 避坑认知 | 发现了潜在的陷阱 |
| "这个术语的意思是..." | 业务认知 | 理解了业务术语的含义 |
| "用户的真实需求其实是..." | 业务认知 | 理解了用户的深层需求 |

### 客开特有触发信号

| 触发信号 | 知识类型 | 示例 |
|----------|----------|------|
| "这两个模块之间居然有依赖" | **系统发现** | 发现了未预期的模块间依赖 |
| "这个修改影响到了其他功能" | **变更影响** | 变更的实际影响超出预期 |
| "这个区域修改风险很高" | **地雷标记** | 识别了高风险修改区域 |
| "这段代码有个隐含的业务规则" | **业务规则** | 从代码中发现了未文档化的业务规则 |
| "这个接口有个隐式契约" | **接口契约** | 发现了接口文档中未记录的约定 |
| "这个数据格式被多处依赖" | **隐式依赖** | 发现了通过数据格式建立的隐式依赖 |

## 客开特有记忆类型详解

### 1. 系统发现

**定义**：在客开过程中发现的关于系统结构的新认知。

**捕获内容**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 发现内容 | 具体发现了什么 | "订单模块通过事件总线通知库存模块" |
| 发现来源 | 在什么场景下发现的 | "在追踪 OrderService.createOrder() 调用链时" |
| 预期外？ | 是否与之前的认知不同 | "是，之前以为订单模块直接调用库存模块" |
| 影响评估 | 这个发现的影响 | "影响范围分析需要考虑事件总线的传播" |

**记录模板**：

```markdown
### 系统发现 - [日期]

**发现内容**：（描述发现的内容）
**发现来源**：（在什么场景下发现的）
**预期外**：（是/否，如果预期外，说明之前的认知是什么）
**影响评估**：（这个发现对后续工作的影响）
**验证状态**：（已验证/待验证）
```

---

### 2. 变更影响

**定义**：变更实施后，实际影响与预期影响的差异。

**捕获内容**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 变更编号 | 对应的变更需求 | CR-003 |
| 预期影响 | 变更前分析的影响范围 | "仅影响订单列表页" |
| 实际影响 | 变更后观察到的实际影响 | "同时影响了订单导出和订单打印" |
| 差异原因 | 为什么实际影响与预期不同 | "导出和打印功能复用了列表的排序逻辑" |
| 认知更新 | 需要更新哪些系统认知 | "系统结构认知中的数据流需要补充" |

**记录模板**：

```markdown
### 变更影响 - [日期] - [变更编号]

**预期影响**：（变更前分析的影响范围）
**实际影响**：（变更后观察到的实际影响）
**差异原因**：（为什么实际影响与预期不同）
**认知更新**：（需要更新哪些系统认知）
**经验教训**：（对后续变更的建议）
```

---

### 3. 地雷标记

**定义**：在客开过程中发现的高风险区域。

**捕获内容**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 地雷区域 | 具体位置 | "PaymentService.process() 方法" |
| 风险描述 | 为什么是高风险 | "涉及多种支付渠道，修改可能影响渠道兼容性" |
| 风险等级 | 高/中/低 | 高 |
| 发现来源 | 如何发现的 | "在分析支付模块时发现" |
| 建议措施 | 如何规避风险 | "修改前必须确认所有支付渠道的测试通过" |

**记录模板**：

```markdown
### 地雷标记 - [日期]

**地雷区域**：（具体位置）
**风险描述**：（为什么是高风险）
**风险等级**：（高/中/低）
**影响范围**：（如果触发，会影响哪些功能）
**发现来源**：（如何发现的）
**建议措施**：（如何规避风险）
```

## 记忆优先级判断

并非所有信息都值得捕获。按以下优先级判断：

| 优先级 | 判断标准 | 处理方式 |
|--------|----------|----------|
| **P0 - 必须捕获** | 地雷区发现、变更影响差异、隐式依赖发现 | 立即记录，立即蒸馏 |
| **P1 - 应该捕获** | 系统结构新发现、业务规则发现、编码模式发现 | 当次交互结束前记录 |
| **P2 - 可以捕获** | 代码风格观察、术语理解、一般性经验 | 阶段蒸馏时统一记录 |
| **P3 - 不需捕获** | 临时性的调试信息、已解决的临时问题 | 不记录 |

## 记忆记录规范

### 记录原则

1. **即时性**：P0 级别的记忆必须即时记录，不能延后
2. **准确性**：记录事实而非推测，推测必须标注
3. **完整性**：记录足够的上下文，使未来的蒸馏能理解这条记忆
4. **可追溯**：记录发现来源和日期，便于后续追溯

### 记录格式

每条记忆碎片应包含以下要素：

```markdown
- **类型**：（系统发现 / 变更影响 / 地雷标记 / 业务规则 / 编码模式）
- **日期**：（YYYY-MM-DD）
- **内容**：（具体内容）
- **来源**：（发现场景）
- **优先级**：（P0/P1/P2）
- **验证状态**：（已验证/推测/待验证）
```

### 存储位置

| 存储位置 | 用途 | 生命周期 |
|----------|------|----------|
| 当前会话上下文 | 临时存储，便于即时引用 | 会话结束前蒸馏 |
| project-brain.md 草稿区 | 等待蒸馏的记忆碎片 | 蒸馏后清除 |
| project-brain.md 正式章节 | 蒸馏后的结构化认知 | 持续保留 |

## 常见遗漏场景

以下场景容易被遗漏，需要特别注意：

| 场景 | 容易遗漏的知识 | 建议 |
|------|----------------|------|
| 修复 Bug 时 | Bug 背后的系统理解 | Bug 修复后，记录 Bug 的根因和涉及的系统知识 |
| 代码审查时 | 审查中发现的隐式依赖 | 审查过程中发现的依赖关系即时记录 |
| 测试失败时 | 测试失败揭示的系统行为 | 测试失败往往揭示了未预期的系统行为 |
| 用户反馈时 | 用户反馈中的业务规则 | 用户反馈可能包含未文档化的业务规则 |
| 配置变更时 | 配置项之间的隐式依赖 | 配置变更可能影响多个模块的行为 |
