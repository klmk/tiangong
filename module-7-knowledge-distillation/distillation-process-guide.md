# 蒸馏流程指南

> **如何将碎片化的记忆提炼为结构化的系统认知。**
>
> 本指南定义了知识蒸馏的标准流程和各阶段的蒸馏重点。

---

## 蒸馏流程总览

```
记忆捕获 → 知识分类 → 事实验证 → 结构化提炼 → 融合更新 → 成长应用
   │           │           │           │           │           │
   ▼           ▼           ▼           ▼           ▼           ▼
 Layer 1    自动分类    确认真实性    提炼核心    更新大脑    优化后续
 自动记忆   归入知识    排除推测    认知要点    文档        行为
```

---

## Step 1: 记忆捕获

在每次交互中，自动识别并记录有价值的知识碎片。

### 客开特有的记忆类型

| 记忆类型 | 识别信号 | 记录方式 |
|----------|----------|----------|
| **系统发现** | 发现了新的模块关系、隐式依赖、数据流路径 | 记录到"系统结构认知"草稿区 |
| **变更影响** | 变更实际影响了预期之外的模块或功能 | 记录到"变更认知"草稿区 |
| **地雷标记** | 发现了高风险区域、历史问题、隐式约束 | 记录到"避坑认知"草稿区 |
| **业务规则** | 发现了代码中隐含的业务规则或约束 | 记录到"业务认知"草稿区 |
| **编码模式** | 发现了代码库中反复出现的特定模式 | 记录到"编码认知"草稿区 |

详见 [`memory-capture-guide.md`](./memory-capture-guide.md)。

---

## Step 2: 知识分类

将捕获的记忆碎片归入对应的知识类型。

### 分类规则

| 如果记忆是关于... | 归入... |
|-------------------|---------|
| 系统的模块关系、架构、数据流 | 系统结构认知 |
| 业务规则、业务流程、业务术语 | 业务认知 |
| 代码风格、设计模式、编码约定 | 编码认知 |
| 已知问题、陷阱、高风险区域 | 避坑认知 |
| 变更历史、变更影响、进行中的变更 | 变更认知 |

---

## Step 3: 事实验证

确保蒸馏的知识基于事实而非推测。

### 验证方法

| 验证方式 | 适用场景 | 说明 |
|----------|----------|------|
| 代码验证 | 技术性知识 | 通过阅读代码确认知识的准确性 |
| 测试验证 | 行为性知识 | 通过运行测试确认行为是否符合预期 |
| 文档验证 | 架构性知识 | 通过查阅文档确认架构描述的准确性 |
| 用户确认 | 业务性知识 | 通过与用户确认业务规则的准确性 |

### 标注规则

- 已验证的事实：正常记录
- 推测但合理：标注 `[推测]` 并说明推测依据
- 不确定：标注 `[待验证]` 并说明需要验证的内容

---

## Step 4: 结构化提炼

将碎片化的知识提炼为结构化的认知要点。

### 提炼原则

1. **去重**：合并重复的知识碎片
2. **抽象**：从具体实例中提炼出通用规律
3. **关联**：建立知识之间的关联关系
4. **分级**：按重要性对知识进行分级（核心 / 重要 / 参考）

### 提炼模板

```
### [知识标题]

**核心认知**：（一句话概括核心认知）

**详细说明**：（展开说明）

**适用范围**：（这个认知在什么范围内适用）

**关联知识**：（与哪些其他知识相关联）

**重要程度**：（核心 / 重要 / 参考）

**发现来源**：（在什么场景下发现的）
```

---

## Step 5: 融合更新

将提炼后的知识更新到 project-brain.md 中。

### 更新规则

1. **新增知识**：直接添加到对应章节
2. **更新知识**：保留变更历史，更新为最新认知
3. **冲突知识**：标注冲突，分析原因，更新为更准确的认知
4. **过时知识**：标记为过时，移到历史记录区（保留一段时间后删除）

### 更新检查清单

- [ ] 系统结构认知是否需要更新？
- [ ] 业务认知是否需要更新？
- [ ] 编码认知是否需要更新？
- [ ] 避坑认知是否需要更新？（特别是地雷区清单）
- [ ] 变更认知是否需要更新？
- [ ] 蒸馏日志是否已记录？

---

## Step 6: 成长应用

基于蒸馏后的知识，优化后续行为。

### 应用方式

| 应用方式 | 说明 | 示例 |
|----------|------|------|
| **影响预测** | 基于系统结构认知，预测变更的影响范围 | "修改订单模块的导出功能，根据系统结构认知，可能影响报表模块" |
| **地雷区预警** | 基于地雷区清单，在变更前主动预警 | "即将修改 PaymentService，根据地雷区清单 MINE-003，需要先确认支付渠道兼容性" |
| **方案优化** | 基于编码认知和业务认知，提供更优的变更方案 | "根据编码模式 CP-002，这个功能应该使用策略模式实现" |
| **风险规避** | 基于避坑认知，主动规避已知风险 | "根据已知问题 KI-005，这个场景下不能使用缓存，需要实时查询" |

## 各阶段蒸馏重点

### Phase 0 完成后：系统结构认知初始化

**蒸馏重点**：
- 系统模块关系是否清晰？
- 核心数据流是否已梳理？
- 技术栈和架构风格是否已确认？
- 关键依赖是否已识别？
- 是否发现了初始的地雷区？

**必更新章节**：第二章（系统结构认知）

---

### Phase 1 完成后：变更需求理解

**蒸馏重点**：
- 变更需求涉及的业务规则是否已理解？
- 影响范围分析是否基于准确的系统结构认知？
- 是否发现了新的业务规则或约束？
- 是否识别了潜在的地雷区？

**必更新章节**：第三章（业务认知）、第六章（变更认知）

---

### Phase 2 完成后：架构理解深化

**蒸馏重点**：
- 是否发现了新的隐式依赖？
- 设计意图是否已理解并记录？
- 是否发现了新的地雷区？
- 系统结构认知是否需要修正？

**必更新章节**：第二章（系统结构认知）、第五章（避坑认知）

---

### Phase 3 完成后：编码模式更新

**蒸馏重点**：
- 是否发现了新的编码模式？
- 实际影响范围与预期是否一致？如果不一致，更新认知
- 是否触发了新的地雷区？
- 遗留代码注意事项是否需要补充？

**必更新章节**：第四章（编码认知）、第五章（避坑认知）、第六章（变更认知）

---

### 每次变更完成后：系统理解更新

> **客开特有蒸馏时机** -- 这是最重要的蒸馏触发点。

**蒸馏重点**：
- **系统理解更新**：变更过程中对系统的理解是否有新的发现？
- **影响范围记录**：变更实际影响了哪些模块？与预期是否一致？
- **地雷区更新**：变更过程中是否发现了新的高风险区域？
- **变更历史记录**：将本次变更记录到变更认知中
- **经验教训**：本次变更中有哪些经验教训值得记录？

**必更新章节**：全部六章（根据实际情况选择性更新）

---

### Phase 5 完成后：全面知识回顾

**蒸馏重点**：
- 回顾整个客开过程中的所有知识积累
- 提炼核心经验教训
- 更新地雷区清单（基于完整的变更经验）
- 规划下一步的知识成长方向
- 评估 AI 对系统的理解深度是否达到预期

**必更新章节**：全部六章 + 蒸馏日志

---

## 蒸馏质量标准

| 标准 | 说明 |
|------|------|
| **准确性** | 知识基于事实，推测已标注 |
| **完整性** | 关键知识无遗漏 |
| **可操作性** | 知识能指导后续行为 |
| **时效性** | 知识反映最新系统状态 |
| **一致性** | 知识之间无冲突 |
