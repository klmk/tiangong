# Module 7: 知识蒸馏（Knowledge Distillation）

> **持续运行的知识蒸馏模块 -- 让 AI 在客开过程中不断成长。**
>
> 与 Paoding 原版共享三层架构（自动记忆 -> 主动蒸馏 -> 持续成长），但增加了"系统理解蒸馏"维度，将客开过程中积累的系统认知持续沉淀为可复用的知识。

---

## 定位

Module 7 是一个**持续运行**的跨阶段模块，贯穿客开全流程（Phase 0-5）。它的核心职责是：

1. **自动记忆**：在每次交互中自动捕获有价值的知识碎片
2. **主动蒸馏**：将碎片化的记忆提炼为结构化的系统认知
3. **持续成长**：通过知识的持续积累，让 AI 对目标系统的理解不断深化

### 与 Paoding 原版的关系

| 维度 | Paoding Module 7 | 客开版 Module 7 |
|------|-----------------|----------------|
| 核心目标 | 沉淀项目知识 | 沉淀系统理解 + 项目知识 |
| 知识类型 | 业务认知、编码认知、避坑认知 | 系统结构认知、业务认知、编码认知、避坑认知、变更认知 |
| 蒸馏触发 | 阶段转换时 | 阶段转换时 + **每次变更完成后** |
| 核心产出 | project-brain.md | project-brain.md（含系统结构认知、地雷区速查） |
| 成长指标 | 项目理解深度 | 系统理解深度 + 变更影响预测准确度 |

## 三层架构

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3: 持续成长 (Continuous Growth)                   │
│  基于蒸馏后的知识，主动优化后续行为                        │
│  - 预测变更影响范围                                      │
│  - 主动识别地雷区                                       │
│  - 优化变更方案建议                                      │
├─────────────────────────────────────────────────────────┤
│  Layer 2: 主动蒸馏 (Active Distillation)                 │
│  将碎片化记忆提炼为结构化认知                             │
│  - 系统结构认知更新                                      │
│  - 地雷区清单更新                                        │
│  - 业务规则更新                                          │
│  - 编码模式更新                                          │
├─────────────────────────────────────────────────────────┤
│  Layer 1: 自动记忆 (Auto Memory Capture)                 │
│  在每次交互中自动捕获知识碎片                             │
│  - 系统发现（新模块关系、隐式依赖）                       │
│  - 变更影响（变更实际影响了哪些模块）                     │
│  - 地雷标记（高风险区域发现）                             │
│  - 业务规则发现                                          │
│  - 编码模式发现                                          │
└─────────────────────────────────────────────────────────┘
```

## 知识类型

### 客开特有知识类型

| 知识类型 | 说明 | 示例 |
|----------|------|------|
| **系统结构认知** | 对系统模块关系、数据流、架构风格的理解 | "订单模块通过事件总线通知库存模块，而非直接调用" |
| **隐式依赖** | 代码中未显式声明但实际存在的依赖关系 | "用户模块的缓存依赖 Redis 中的 session key 格式" |
| **地雷区域** | 高风险修改区域、历史问题区域 | "PaymentService.process() 修改前必须确认所有支付渠道的兼容性" |
| **业务规则** | 已有系统中的业务规则和约束 | "订单金额超过 10000 时需要走审批流程" |
| **接口契约** | 模块间接口的隐式契约和约定 | "所有 API 返回的时间字段都是 UTC 毫秒时间戳" |

### 通用知识类型（与 Paoding 相同）

| 知识类型 | 说明 |
|----------|------|
| 业务认知 | 业务领域知识、术语、流程 |
| 编码认知 | 代码风格、设计模式、编码规范 |
| 避坑认知 | 已知问题、常见陷阱、错误模式 |

## project-brain.md 核心章节

project-brain.md 是知识蒸馏的核心产出物，包含以下六大认知章节：

| 章节 | 说明 | 更新频率 |
|------|------|----------|
| 1. AI身份锚定 | AI 的角色定位和工作原则 | 项目初始化时设定 |
| 2. 系统结构认知 | 模块关系、核心数据流、技术栈、架构风格、关键依赖 | 每次变更后更新 |
| 3. 业务认知 | 业务领域知识、术语、已有业务规则 | 发现新规则时更新 |
| 4. 编码认知 | 代码风格、设计模式、代码库特定模式、遗留代码注意事项 | 发现新模式时更新 |
| 5. 避坑认知 | 已知问题、常见陷阱、**地雷区清单** | 发现新陷阱时更新 |
| 6. 变更认知 | 进行中的变更、已完成变更历史、已知问题、下一步计划 | 每次变更后更新 |

详见 [`project-brain-template.md`](./project-brain-template.md)。

## 6 条铁律

### 铁律一：每次变更完成后必须蒸馏
> 变更完成后，必须执行至少一次知识蒸馏。变更中获得的系统理解、发现的地雷区、实际影响范围，必须及时沉淀。不蒸馏的变更等于浪费了一次学习机会。

### 铁律二：事实优先于推测
> 蒸馏的知识必须基于实际观察和验证的事实，而非推测。如果不确定，标注为"待验证"而非当作事实记录。

### 铁律三：知识必须可操作
> 沉淀的知识必须能指导后续行为。如果一条知识无法转化为具体的行动指导，它要么需要进一步提炼，要么不值得沉淀。

### 铁律四：冲突知识必须标注
> 当新发现与已有知识冲突时，必须明确标注冲突，分析冲突原因，并更新为更准确的认知。不能让冲突知识共存而不处理。

### 铁律五：地雷区必须记录
> 在客开过程中发现的任何高风险区域、隐式依赖、历史问题区域，必须记录到地雷区清单中。遗漏地雷区记录可能导致后续变更引发事故。

### 铁律六：project-brain.md 是活的文档
> project-brain.md 不是一次性的产出物，而是持续更新的活文档。每次蒸馏后都必须更新 project-brain.md，确保它始终反映最新的系统认知。

## 蒸馏时机

| 时机 | 蒸馏重点 | 必须执行 |
|------|----------|----------|
| Phase 0 完成后 | 系统结构认知初始化、技术栈记录、架构风格确认 | 是 |
| Phase 1 完成后 | 变更需求理解、影响范围认知、业务规则发现 | 是 |
| Phase 2 完成后 | 架构理解深化、隐式依赖发现、设计意图记录 | 是 |
| Phase 3 完成后 | 编码模式更新、地雷区标记、实际影响范围记录 | 是 |
| **每次变更完成后** | 系统理解更新、影响范围记录、地雷区更新、变更历史记录 | **是** |
| Phase 5 完成后 | 全面知识回顾、经验教训提炼、下一步计划更新 | 是 |
| 发现重要信息时 | 即时捕获，不等待阶段转换 | 建议 |
