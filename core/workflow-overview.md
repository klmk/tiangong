# TianGong 全流程总览 (Workflow Overview)

> **客开场景的 6 阶段开发流程，从系统理解到变更交付。**
>
> 核心理念：先理解，后动手；先保护，后改造；先验证，后交付。

---

## 流程全景

```
Phase 0 ──→ Phase 1 ──→ Phase 2 ──→ Phase 3 ──→ Phase 4 ──→ Phase 5
系统画像    变更分析    架构理解    变更实施    知识沉淀    变更交付
  │            │            │            │            │            │
  │            │            │            │            │            │
  ▼            ▼            ▼            ▼            ▼            ▼
[硬性门控]  [硬性门控]   [硬性门控]   [硬性门控]   [软性门控]  [硬性门控]
 G-0         G-1          G-2          G-3          G-4          G-5
```

---

## Phase 0 -- 系统画像

> **"在改造一个系统之前，先理解这个系统。"**

### 目标

建立对目标系统的全局认知，形成系统画像，为后续的变更分析提供基础。

### 核心活动

| 活动 | 说明 |
|---|---|
| 系统结构扫描 | 识别系统的模块划分、目录结构、入口文件 |
| 技术栈识别 | 确认编程语言、框架、中间件、数据库等技术选型 |
| 关键模块标注 | 识别核心业务模块、公共基础模块、配置模块 |
| 依赖关系梳理 | 梳理模块间的依赖关系和数据流向 |
| 关键约束记录 | 记录系统的非功能性约束（性能、安全、兼容性等） |
| 现有测试评估 | 评估现有测试的覆盖情况和质量 |

### 必输出物

| 输出物 | 格式 | 说明 |
|---|---|---|
| 系统画像文档 | Markdown | 系统结构图、模块清单、技术栈、关键约束 |
| 模块依赖图 | 文本/图表 | 模块间的依赖关系和数据流向 |
| 测试覆盖评估 | Markdown | 现有测试的覆盖范围和质量评估 |

### 门控 G-0（硬性）

| 检查项 | 通过标准 |
|---|---|
| 系统结构是否清晰 | 能画出系统的模块划分和依赖关系 |
| 技术栈是否明确 | 所有关键技术选型已识别并记录 |
| 关键约束是否记录 | 系统的非功能性约束已识别并记录 |

---

## Phase 1 -- 变更分析

> **"在动手之前，先想清楚要改什么、为什么改、改了会怎样。"**

### 目标

将用户的变更需求转化为清晰、完整、可验证的变更规格，明确影响范围和回归风险。

### 核心活动

| 活动 | 说明 |
|---|---|
| 需求澄清 | 将用户的变更需求转化为结构化的变更描述 |
| 影响范围分析 | 识别变更会影响哪些模块、接口、数据 |
| 回归风险评估 | 评估变更可能破坏哪些已有功能 |
| 验收标准定义 | 明确变更完成的可验证标准 |
| 优先级排序 | 当有多个变更需求时，确定实施顺序 |
| 风险缓解计划 | 针对高风险变更制定缓解措施 |

### 必输出物

| 输出物 | 格式 | 说明 |
|---|---|---|
| 变更分析文档 | Markdown | 变更描述、影响范围、回归风险、验收标准 |
| 影响范围矩阵 | 表格 | 变更点与受影响模块/功能的映射关系 |
| 回归风险清单 | 表格 | 识别的回归风险及对应的缓解措施 |

### 门控 G-1（硬性）

| 检查项 | 通过标准 |
|---|---|
| 变更描述是否清晰 | 变更的具体内容、预期行为、边界条件均已明确 |
| 影响范围是否完整 | 所有受影响的模块和功能已被识别 |
| 回归风险是否评估 | 所有已识别的风险有对应的缓解措施 |
| 验收标准是否可验证 | 验收标准是具体的、可测试的、可度量的 |
| 用户是否确认 | 用户已确认变更分析文档的内容 |

---

## Phase 2 -- 架构理解

> **"不理解现有架构，就不可能安全地改造它。"**

### 目标

深入理解变更涉及的相关模块的架构设计、业务逻辑和数据流，为安全实施变更提供知识基础。

### 核心活动

| 活动 | 说明 |
|---|---|
| 相关模块深度分析 | 分析变更涉及的所有模块的内部结构 |
| 业务逻辑梳理 | 理解相关模块的业务规则和处理流程 |
| 数据流追踪 | 追踪数据在模块间的流转路径 |
| 设计意图推断 | 理解现有代码的设计决策和背后的原因 |
| 隐含约束识别 | 识别代码中未显式声明但实际存在的约束 |
| 变更方案设计 | 基于架构理解，设计具体的变更实施方案 |

### 必输出物

| 输出物 | 格式 | 说明 |
|---|---|---|
| 架构理解文档 | Markdown | 相关模块的设计逻辑、数据流、依赖关系 |
| 变更方案文档 | Markdown | 具体的变更实施方案、变更步骤、风险评估 |
| 回归测试计划 | Markdown | 需要编写的回归测试清单和优先级 |

### 门控 G-2（硬性）

| 检查项 | 通过标准 |
|---|---|
| 相关模块是否理解 | 能清晰描述变更涉及模块的设计逻辑和数据流 |
| 设计意图是否明确 | 理解现有代码为什么是这样实现的 |
| 隐含约束是否识别 | 代码中的隐含约束已被识别并记录 |
| 变更方案是否可行 | 变更方案经过影响分析验证，风险可控 |
| 回归测试计划是否完整 | 所有受影响功能都有对应的回归测试计划 |

---

## Phase 3 -- 变更实施

> **"在安全网的守护下，精确地实施变更。"**

### 目标

按照变更方案，在回归测试的保护下，安全地实施代码变更。

### 核心活动

| 活动 | 说明 |
|---|---|
| 回归测试编写 | 按照回归测试计划，编写并确认回归测试 |
| 代码变更实施 | 按照变更方案，逐步实施代码变更 |
| 变更自审 | 对每一步变更进行自我审查 |
| 测试执行 | 运行回归测试和新增功能测试 |
| 问题修复 | 修复测试发现的问题 |
| 变更记录 | 记录每一步变更的内容和原因 |

### 必输出物

| 输出物 | 格式 | 说明 |
|---|---|---|
| 回归测试代码 | 源代码 | 覆盖受影响功能的自动化回归测试 |
| 变更后的代码 | 源代码 | 经过测试验证的变更代码 |
| 变更记录文档 | Markdown | 每一步变更的内容、原因、对应需求编号 |
| 测试报告 | Markdown | 回归测试和新增测试的执行结果 |

### 门控 G-3（硬性）

| 检查项 | 通过标准 |
|---|---|
| 回归测试是否通过 | 所有回归测试用例通过 |
| 新功能测试是否通过 | 所有新增功能测试用例通过 |
| 变更是否可追溯 | 每段变更代码可追溯到变更需求 |
| 变更是否经过审查 | 变更已按审查维度逐项检查 |
| 原有功能是否未受影响 | 受影响功能的验证结果确认无回归问题 |

---

## Phase 4 -- 知识沉淀

> **"每一次变更都是加深系统理解的机会。"**

### 目标

将变更过程中获得的知识和经验沉淀下来，更新系统文档，为后续变更提供更好的基础。

### 核心活动

| 活动 | 说明 |
|---|---|
| 系统画像更新 | 根据变更结果更新系统画像文档 |
| 经验教训总结 | 总结变更过程中的经验教训和最佳实践 |
| 技术债务记录 | 记录变更过程中发现的技术债务 |
| 知识库更新 | 更新系统知识库，补充新获得的系统理解 |
| 流程改进建议 | 提出框架流程的改进建议 |

### 必输出物

| 输出物 | 格式 | 说明 |
|---|---|---|
| 更新后的系统画像 | Markdown | 反映变更后系统状态的系统画像 |
| 经验教训文档 | Markdown | 本次变更的经验教训和改进建议 |
| 技术债务清单 | 表格 | 变更过程中发现的技术债务及建议处理方式 |

### 门控 G-4（软性）

| 检查项 | 通过标准 |
|---|---|
| 系统画像是否更新 | 系统画像反映了变更后的最新状态 |
| 经验教训是否记录 | 关键经验教训已被记录 |
| 技术债务是否登记 | 发现的技术债务已被登记 |

> **说明**：此门控为软性门控。如果时间紧张，可以简化知识沉淀的深度，但不能完全跳过。至少要更新系统画像和记录技术债务。

---

## Phase 5 -- 变更交付

> **"交付的不是代码，而是经过验证的、可追溯的、有文档保障的变更。"**

### 目标

将变更成果以完整、可交付的形式呈现，确保用户能够理解变更内容、验证变更结果、部署变更。

### 核心活动

| 活动 | 说明 |
|---|---|
| 变更总结 | 汇总本次变更的完整内容 |
| 部署说明编写 | 编写变更的部署步骤和注意事项 |
| 回退方案编写 | 编写变更的回退方案 |
| 最终验证 | 执行最终的全面验证 |
| 交付确认 | 与用户确认交付内容 |

### 必输出物

| 输出物 | 格式 | 说明 |
|---|---|---|
| 变更交付文档 | Markdown | 变更总结、变更清单、影响范围 |
| 部署说明 | Markdown | 部署步骤、前置条件、注意事项 |
| 回退方案 | Markdown | 回退步骤、回退条件、回退验证 |
| 最终验证报告 | Markdown | 最终验证的完整结果 |

### 门控 G-5（硬性）

| 检查项 | 通过标准 |
|---|---|
| 变更内容是否完整 | 所有计划的变更均已实施并验证 |
| 部署说明是否可执行 | 部署步骤清晰、完整、可操作 |
| 回退方案是否可行 | 回退方案经过验证，可以在需要时执行 |
| 最终验证是否通过 | 所有验证项通过，原有功能未受影响 |
| 用户是否确认交付 | 用户已确认交付内容符合预期 |

## 门控体系总览

| 门控 | 阶段 | 类型 | 核心关注点 |
|---|---|---|---|
| G-0 | Phase 0 → Phase 1 | **硬性** | 系统理解是否充分 |
| G-1 | Phase 1 → Phase 2 | **硬性** | 变更需求是否清晰完整 |
| G-2 | Phase 2 → Phase 3 | **硬性** | 架构理解是否到位，方案是否可行 |
| G-3 | Phase 3 → Phase 4 | **硬性** | 变更是否安全，测试是否通过 |
| G-4 | Phase 4 → Phase 5 | **软性** | 知识是否沉淀（可简化但不能跳过） |
| G-5 | Phase 5 → 交付 | **硬性** | 交付物是否完整，验证是否通过 |

### 硬性门控 vs 软性门控

- **硬性门控**：必须全部通过才能进入下一阶段。任何一项不通过，必须回退处理。
- **软性门控**：建议全部通过，但在时间紧张等特殊情况下，可以简化处理。简化处理必须在变更记录中说明原因。

## 回退机制

当在某个阶段发现问题或不符合门控条件时，按以下规则回退：

```
                    ┌─────────────────────────────┐
                    │                             │
                    ▼                             │
Phase 0 ──→ Phase 1 ──→ Phase 2 ──→ Phase 3 ──→ Phase 4 ──→ Phase 5
系统画像    变更分析    架构理解    变更实施    知识沉淀    变更交付
  │            │            │            │            │            │
  │            │            │            │            │            │
  │    ┌───────┘            │            │            │            │
  │    │  需求不清晰 ───────┘            │            │            │
  │    │  回退到 Phase 1               │            │            │
  │    │                               │            │            │
  │    │         ┌─────────────────────┘            │            │
  │    │         │  架构理解不足 ───────────────────┘            │
  │    │         │  回退到 Phase 2                            │
  │    │         │                                            │
  │    │         │          ┌─────────────────────────────────┘
  │    │         │          │  测试失败/变更有误 ───────────────┘
  │    │         │          │  回退到 Phase 3
  │    │         │          │
  │    │         │          │
  └────┘         └──────────┘
  系统画像不准确
  回退到 Phase 0
```

### 回退规则详表

| 触发条件 | 回退目标 | 处理要求 |
|---|---|---|
| Phase 1 发现系统理解有误 | Phase 0 | 重新进行系统画像，修正错误认知 |
| Phase 2 发现变更需求不清晰 | Phase 1 | 重新澄清需求，更新变更分析文档 |
| Phase 3 发现架构理解不足 | Phase 2 | 深入分析相关模块，更新架构理解文档 |
| Phase 3 测试失败 | Phase 3（当前阶段内） | 修复问题，重新测试。不需要回退到前一阶段 |
| Phase 3 发现变更方案不可行 | Phase 2 | 重新设计变更方案 |
| Phase 4 发现遗留问题 | Phase 3 | 回退修复遗留问题 |
| Phase 5 最终验证失败 | Phase 3 | 回退到变更实施阶段，修复问题 |

### 回退原则

1. **尽早回退**：发现问题后立即回退，不要试图"绕过"问题继续推进
2. **回退到根源**：回退到问题产生的阶段，而非仅回退一个阶段
3. **记录回退原因**：每次回退都必须记录原因和发现的问题
4. **更新文档**：回退后必须更新相关文档，反映新的认知

## 与 Paoding 原版的对比

### 核心差异

| 维度 | Paoding（Greenfield） | TianGong（Brownfield） |
|---|---|---|
| **阶段数** | 5 个阶段 | 6 个阶段（增加了 Phase 0 系统画像） |
| **起始点** | 需求分析 | 系统画像（先理解系统） |
| **核心风险** | 需求理解偏差 | 破坏已有功能 |
| **测试策略** | 新功能测试为主 | 回归测试为主 |
| **门控重点** | 设计完整性和代码质量 | 影响分析和回归安全 |
| **文档重点** | 设计文档和 API 文档 | 变更文档和影响分析 |
| **AI 角色** | 创造者 | 理解者 + 改造者 |

### 阶段对应关系

```
Paoding                    TianGong
─────────                  ──────────
                           Phase 0: 系统画像 (新增)
Phase 1: 需求分析    ←→    Phase 1: 变更分析
Phase 2: 系统设计    ←→    Phase 2: 架构理解
Phase 3: 编码实现    ←→    Phase 3: 变更实施
Phase 4: 测试验证    ←→    (融入 Phase 3)
                           Phase 4: 知识沉淀 (新增)
Phase 5: 交付部署    ←→    Phase 5: 变更交付
```

### 关键区别说明

1. **新增 Phase 0（系统画像）**：Paoding 从需求开始，因为是从零构建。客开场景必须先理解现有系统，否则无法评估变更的影响。系统画像是客开流程的基石。

2. **Phase 1 从"需求分析"变为"变更分析"**：客开的需求分析重点是"变更"而非"全新需求"。核心是分析变更的影响范围和回归风险，而非从零设计功能。

3. **Phase 2 从"系统设计"变为"架构理解"**：Paoding 的设计是"创造"新架构，客开的重点是"理解"现有架构。设计活动在 Phase 2 中仍然存在（变更方案设计），但前提是充分理解现有架构。

4. **测试融入 Phase 3**：Paoding 的测试是独立阶段，因为新功能测试可以在编码完成后集中进行。客开场景中，回归测试必须在编码之前编写（IL-2），因此测试活动融入变更实施阶段。

5. **新增 Phase 4（知识沉淀）**：每次变更都是加深系统理解的机会。Paoding 不需要这个阶段，因为系统是从零构建的。客开场景中，知识沉淀能持续提升后续变更的效率和质量。

6. **门控体系更严格**：Paoding 的门控侧重设计完整性和代码质量。客开的门控侧重影响分析和回归安全，因为客开的最大风险是破坏已有功能。

---

## 流程执行原则

1. **顺序执行**：6 个阶段按顺序执行，不得跳过任何阶段
2. **门控阻断**：硬性门控不通过时，必须回退处理，不得强行推进
3. **可简化不可跳过**：用户可以要求简化某个阶段的深度，但不能跳过整个阶段
4. **持续验证**：每个阶段都应包含验证活动，而非仅在 Phase 3 和 Phase 5 验证
5. **文档同步**：文档在对应阶段同步输出，不得延后补写
