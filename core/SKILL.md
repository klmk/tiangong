# TianGong SKILL.md

> **天工（TianGong）** -- 面向已有系统定制化开发的 AI 驱动框架
>
> 基于庖丁框架改造，核心区别：Paoding 从无到有（Greenfield），本框架面向已有系统的定制化改造（Brownfield Customization）。

---

## 一、技能自动触发规则

### 1% 原则

当用户输入满足以下**任一条件**时，本技能自动激活：

| 触发条件 | 示例 |
|---|---|
| 明确提到对**已有系统/模块**进行修改、扩展、适配 | "帮我在现有的订单模块里加一个审批流程" |
| 提到**客开、定制化、二次开发、改造**等关键词 | "这个客户需要定制一个导出功能" |
| 要求**理解或分析**现有代码后再做变更 | "先看看这个模块的逻辑，然后加个字段" |
| 涉及**回归、兼容、影响范围**等客开特有概念 | "改了这个接口会不会影响其他功能" |
| 用户提供了一个**变更需求**而非全新需求 | "需求是把现在的列表改成支持分页" |
| 用户要求**修改、修复、优化**已有功能 | "这个查询太慢了，优化一下" |

**不触发条件**（应使用 Paoding 原版或其他技能）：

- 全新项目从零开始搭建
- 纯技术学习、知识问答
- 与代码变更无关的通用任务

## 二、四级优先级体系

本框架中，所有决策和行为遵循以下优先级，**高优先级覆盖低优先级**：

```
┌─────────────────────────────────────────┐
│  Level 1: 铁律 (Iron Laws)              │  ← 不可违反，违反即停止
├─────────────────────────────────────────┤
│  Level 2: 用户指令 (User Commands)       │  ← 用户明确要求，优先执行
├─────────────────────────────────────────┤
│  Level 3: 框架技能 (Framework Skills)    │  ← 本框架定义的流程与方法
├─────────────────────────────────────────┤
│  Level 4: 默认行为 (Default Behavior)    │  ← AI 的通用能力与习惯
└─────────────────────────────────────────┘
```

### 各级说明

**Level 1 -- 铁律**
- 来源：`iron-laws.md` 中定义的 6 条铁律
- 效力：绝对约束，任何情况下不得违反
- 违反后果：立即停止当前操作，回退到最近的安全状态
- 与其他级别冲突时：铁律始终胜出

**Level 2 -- 用户指令**
- 用户明确表达的、具体的操作要求
- 示例："只改这个文件，不要动其他文件"
- 约束：不得违反铁律（Level 1）；若用户指令与铁律冲突，必须**先警告用户**并获得确认

**Level 3 -- 框架技能**
- 本框架定义的标准流程、方法、模板
- 来源：本目录下的所有 `.md` 文件
- 效力：在没有更高优先级指令时，按框架流程执行

**Level 4 -- 默认行为**
- AI 模型的通用能力和默认工作方式
- 仅在以上三级均未覆盖时生效

## 三、"精神与字面"原则

### 核心表述

> **遵守精神，而非仅仅遵守字面。**

### 含义

1. **字面遵守**：机械地按照文档字面描述执行，但不理解其背后的意图
2. **精神遵守**：理解每条规则背后的**目的和原理**，在具体场景中做出正确的判断

### 实践要求

- 当规则的字面表述与具体场景存在模糊地带时，以**规则的意图**为准
- 当发现规则未覆盖的边界情况时，按**规则的精神**推导合理的处理方式
- 主动识别用户指令中可能隐含的意图，而非仅按字面执行

### 示例

| 场景 | 字面遵守 | 精神遵守 |
|---|---|---|
| 用户说"改一下这个函数" | 直接修改函数代码 | 先理解函数的调用链和影响范围，确认变更安全后再改 |
| 用户说"这个很简单" | 跳过分析直接改 | 仍然进行基本的影响分析，只是可以适当简化深度 |
| 铁律要求"回归测试" | 写一个形式上的测试 | 编写真正能捕获回归问题的测试用例 |

## 四、客开场景的特殊说明

### AI 的角色定位

> 在客开场景中，AI 的角色是 **"理解者 + 改造者"**，而非 "创造者"。

这一角色定位决定了本框架与 Paoding 原版的核心差异：

| 维度 | Paoding（Greenfield） | TianGong（Brownfield） |
|---|---|---|
| **起点** | 空白画布 | 已有系统 |
| **首要任务** | 创造新功能 | 理解现有系统 |
| **最大风险** | 需求理解偏差 | 破坏已有功能 |
| **核心能力** | 设计与构建 | 分析与改造 |
| **质量标准** | 功能正确性 | 功能正确性 + 向后兼容性 |
| **测试重点** | 新功能测试 | 回归测试 > 新功能测试 |
| **文档重点** | 设计文档 | 变更文档 + 影响分析 |

### 三大客开原则

1. **先理解，后动手**：不理解现有代码的逻辑和约束，不做任何变更
2. **先保护，后改造**：没有回归测试保护，不动任何已有代码
3. **先验证，后交付**：变更完成后，必须验证原有功能不受影响

### 客开思维模型

```
        ┌──────────┐
        │ 变更需求  │
        └────┬─────┘
             │
             ▼
   ┌─────────────────┐
   │  这个变更会影响  │──→ 影响范围分析
   │  哪些现有功能？  │
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │  现有代码的约束  │──→ 理解现有逻辑
   │  和设计意图是？  │
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │  如何安全地实施  │──→ 制定变更方案
   │  这个变更？     │
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │  如何验证原有   │──→ 回归测试策略
   │  功能未受影响？  │
   └─────────────────┘
```

## 五、生效与约束条款

### 生效范围

- 本技能在**客开（Customization）场景**下自动激活
- 激活后，铁律（Level 1）立即生效
- 框架流程（Level 3）在用户未明确指定其他流程时生效

### 约束条款

1. **铁律不可协商**：6 条铁律在任何情况下均不可跳过、简化或绕过
2. **用户可调整流程深度**：用户可以要求简化框架流程的深度，但不能跳过铁律
3. **AI 有义务提醒**：当检测到用户指令可能违反铁律时，AI 必须明确提醒
4. **持续改进**：本框架的规则集应随着实践不断优化，但修改铁律需要充分论证

### 版本信息

| 项目 | 值 |

| 基于框架 | Paoding（庖丁） |
| 版本 | 1.0 |
