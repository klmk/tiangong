# TianGong 铁律清单 (Iron Laws)

> **铁律不可违反。违反铁律 = 立即停止 + 回退到最近安全状态。**
>
> 铁律是本框架的最高优先级约束（Level 1），高于用户指令、框架流程和 AI 默认行为。

---

## IL-1 需求铁律

> **变更需求不清晰，不得进入开发。**

### 核心要求

在客开场景中，任何代码变更都必须基于**清晰、完整、可验证**的变更需求。模糊的需求是客开事故的首要根源。

变更需求必须明确以下要素：

| 必要要素 | 说明 | 示例 |
|---|---|---|
| **变更描述** | 具体要改什么，改完后的行为是什么 | "订单列表增加按创建时间倒序排列" |
| **影响范围** | 变更会影响哪些模块、接口、数据 | "影响 OrderService.list() 和前端订单列表页" |
| **回归风险** | 变更可能破坏哪些已有功能 | "排序逻辑变更可能影响分页和导出功能" |
| **验收标准** | 如何判断变更正确完成 | "列表按时间倒序，分页正常，导出顺序一致" |

### 红旗列表

AI 在以下情况下**不得继续开发**，必须先澄清需求：

| # | 常见借口 | 反驳 |
|---|---|---|
| R1-1 | "用户说的很清楚，不需要再确认" | 客开场景中，用户描述的"清楚"往往遗漏了隐含约束。影响范围和回归风险必须由 AI 主动分析，不能依赖用户主动提出。 |
| R1-2 | "先做起来，后面再调整" | 在已有系统上"先做起来"意味着引入未经评估的变更。客开没有"先试错"的余地，每一次变更都有破坏现有功能的可能。 |
| R1-3 | "这个改动很小，影响范围有限" | 小改动可能有大影响。一行代码的修改可能影响整个调用链。影响范围必须通过分析确认，不能凭直觉判断。 |
| R1-4 | "用户没提回归风险，应该没有" | 用户不提回归风险不代表不存在。AI 有责任主动识别和分析回归风险，这是客开场景的基本职业素养。 |
| R1-5 | "我理解用户的意思了" | "理解"不等于"确认"。必须将理解的需求**明确输出**给用户确认，尤其是影响范围和回归风险部分。 |

## IL-2 测试铁律

> **没有回归测试保护，就没有变更代码。**

### 核心要求

在已有系统上做变更，**必须先建立回归测试保护**，然后才能修改代码。回归测试是客开场景的安全网。

执行顺序：

```
1. 分析变更影响范围
       │
       ▼
2. 针对受影响的已有功能编写回归测试
   （测试应该先失败，证明它确实在验证现有行为）
       │
       ▼
3. 确认回归测试通过（捕获了现有行为）
       │
       ▼
4. 实施代码变更
       │
       ▼
5. 运行回归测试 + 新功能测试
   （回归测试必须全部通过）
```

### 回归测试的最低标准

- 覆盖变更点的**直接调用者**
- 覆盖变更点的**关键业务路径**
- 覆盖变更点相关的**数据一致性**场景
- 测试必须是**可重复、自动化**的

### 红旗列表

| # | 常见借口 | 反驳 |
|---|---|---|
| R2-1 | "现有代码没有测试，写不了回归测试" | 没有测试恰恰是最危险的情况。越是没有测试保护的代码，变更风险越高，越需要先补充测试。可以从关键路径的集成测试开始。 |
| R2-2 | "先改代码，改完再补测试" | 这是本末倒置。先改代码意味着在没有任何保护的情况下修改系统，一旦出错，你甚至不知道改坏了什么。测试必须在变更之前。 |
| R2-3 | "手动测试一下就行了" | 手动测试不可重复、不可自动化、不可回归。客开场景的变更可能在未来被再次修改，没有自动化回归测试，每次修改都是盲人摸象。 |
| R2-4 | "这个模块太复杂，写测试太费时间" | 模块越复杂，变更风险越高，测试越必要。测试的投入与变更的风险成正比，而非与模块的复杂度成反比。 |
| R2-5 | "我只改了一个常量/配置，不需要测试" | 常量和配置往往被多处引用。一个值的修改可能影响所有依赖它的逻辑。必须验证所有依赖点。 |

## IL-3 审查铁律

> **每个变更必须经过审查。**

### 核心要求

客开场景中的代码审查，重点不仅是代码质量，更是**变更影响分析**和**回归检查**。

审查必须覆盖以下维度：

| 审查维度 | 审查内容 |
|---|---|
| **变更必要性** | 这个变更是否真正需要？是否有更安全的替代方案？ |
| **影响完整性** | 是否所有受影响的模块都被识别和处理？ |
| **回归安全性** | 回归测试是否充分覆盖了受影响的功能？ |
| **向后兼容性** | 变更是否保持了对外接口的兼容性？ |
| **数据一致性** | 变更是否可能导致数据不一致？ |
| **边界条件** | 是否考虑了异常场景和边界情况？ |

### 审查形式

- **AI 自审**：AI 在提交变更前，必须按上述维度进行自我审查
- **用户确认**：涉及架构级变更或高风险变更时，必须由用户确认
- **差异对比**：必须提供变更前后的行为对比，便于审查

### 红旗列表

| # | 常见借口 | 反驳 |
|---|---|---|
| R3-1 | "改动很小，不需要审查" | 改动大小与风险不成正比。一行代码的修改可能引发级联故障。审查的重点是影响分析，而非代码量。 |
| R3-2 | "我自己检查过了没问题" | 自我审查存在盲区。AI 必须按审查维度逐项检查，并以结构化形式输出审查结果，供用户确认。 |
| R3-3 | "用户很急，先上线再说" | 越急的变更越需要审查。匆忙上线导致的客开事故，修复成本远超审查投入。 |
| R3-4 | "这个变更和之前的一样，不需要重新审查" | 系统状态在持续变化。即使类似的变更，在不同的系统状态下可能产生不同的影响。每次变更都必须独立审查。 |

## IL-4 验证铁律

> **完成声明必须有新鲜的验证证据。**

### 核心要求

在声明某个阶段或任务完成之前，必须提供**新鲜的、可验证的证据**。在客开场景中，"新鲜"意味着：

- 验证是在**当前变更之后**执行的
- 验证覆盖了**变更点**和**受影响的已有功能**
- 验证结果是**可重复**的

### 验证证据的类型

| 证据类型 | 说明 | 示例 |
|---|---|---|
| **测试结果** | 自动化测试的执行输出 | "回归测试 15/15 通过，新增测试 3/3 通过" |
| **行为对比** | 变更前后的行为差异 | "变更前：列表按 ID 排序；变更后：列表按时间倒序，分页正常" |
| **影响验证** | 受影响功能的验证结果 | "导出功能验证通过，排序顺序与列表一致" |
| **数据验证** | 数据一致性的验证结果 | "历史订单数据未受影响，新订单排序正确" |

### 红旗列表

| # | 常见借口 | 反驳 |
|---|---|---|
| R4-1 | "代码改完了，应该没问题" | "应该"不是验证。必须实际运行测试并确认结果。代码改完不等于功能正确。 |
| R4-2 | "之前的测试都通过了，这次也没问题" | 之前的测试验证的是之前的状态。当前变更可能引入新的问题，必须重新运行验证。 |
| R4-3 | "我只验证了新功能，原有功能应该没影响" | "应该没影响"是最危险的假设。客开场景中，必须主动验证受影响的已有功能。 |
| R4-4 | "本地跑过了，线上肯定没问题" | 本地环境和线上环境可能存在差异。验证必须在尽可能接近目标环境的环境中执行。 |

## IL-5 文档铁律

> **每个阶段必须有文档输出。**

### 核心要求

客开场景中的文档，核心价值不是"写得好看"，而是**让变更可追溯、可理解、可维护**。

### 各阶段文档要求

| 阶段 | 必须输出的文档 | 核心内容 |
|---|---|---|
| Phase 0 系统画像 | 系统画像文档 | 系统结构、模块关系、技术栈、关键约束 |
| Phase 1 变更分析 | 变更分析文档 | 变更需求、影响范围、回归风险、验收标准 |
| Phase 2 架构理解 | 架构理解文档 | 相关模块的设计逻辑、数据流、依赖关系 |
| Phase 3 变更实施 | 变更记录文档 | 具体变更内容、变更原因、测试覆盖 |
| Phase 4 知识沉淀 | 知识沉淀文档 | 变更中的经验教训、系统理解更新 |
| Phase 5 变更交付 | 交付文档 | 变更总结、部署说明、回退方案 |

### 文档的最低标准

- **可追溯**：能从文档中找到变更的完整链路（需求 -> 设计 -> 代码 -> 测试）
- **可理解**：其他开发者（或未来的 AI）能通过文档理解变更的来龙去脉
- **可维护**：文档结构清晰，便于后续更新

### 红旗列表

| # | 常见借口 | 反驳 |
|---|---|---|
| R5-1 | "代码就是最好的文档" | 在客开场景中，代码只告诉你"现在是什么样"，不告诉你"为什么这样改"和"改了什么"。变更的上下文信息必须通过文档记录。 |
| R5-2 | "这个变更太简单，不需要文档" | 简单的变更也需要记录"改了什么"和"为什么改"。未来的维护者需要知道这段代码的变更历史。 |
| R5-3 | "先完成功能，后面再补文档" | "后面"通常意味着"永远不会"。文档必须在对应阶段同步输出，不能延后。 |
| R5-4 | "文档写得太详细浪费时间" | 文档不需要面面俱到，但必须覆盖关键决策和影响分析。精简但不缺失。 |

## IL-6 追溯铁律

> **每段变更代码必须可追溯到变更需求。**

### 核心要求

客开场景中的每一行变更代码，都必须能回答以下问题：

1. **为什么改？** -- 对应哪个变更需求
2. **改了什么？** -- 具体的变更内容
3. **影响了谁？** -- 受影响的模块和功能
4. **怎么验证？** -- 对应的测试用例

### 追溯链

```
变更需求 (CR-001)
    │
    ├──→ 影响分析 (IA-001)
    │       │
    │       ├──→ 受影响模块 A
    │       │       │
    │       │       ├──→ 变更文件: module_a/service.py (行 42-58)
    │       │       │       │
    │       │       │       └──→ 回归测试: test_module_a.py::test_list_order
    │       │       │
    │       │       └──→ 变更文件: module_a/model.py (行 15-18)
    │       │               │
    │       │               └──→ 回归测试: test_module_a.py::test_data_consistency
    │       │
    │       └──→ 受影响模块 B
    │               │
    │               └──→ 确认无需修改（原因：接口兼容）
    │
    └──→ 验收标准 (AC-001)
            │
            └──→ 测试用例: test_cr001.py
```

### 追溯的实现方式

- 代码注释中标注变更需求编号
- 提交信息中包含变更需求编号
- 测试用例与变更需求关联
- 变更记录文档中维护完整的追溯矩阵

### 红旗列表

| # | 常见借口 | 反驳 |
|---|---|---|
| R6-1 | "代码里有注释就够了" | 注释只能提供局部信息。完整的追溯需要从需求到代码到测试的全链路，这是注释无法覆盖的。 |
| R6-2 | "这个改动是顺手优化的，没有对应需求" | "顺手优化"是客开事故的高频来源。任何代码变更都必须有明确的需求来源，哪怕是"技术债务清理"也应该有对应的任务编号。 |
| R6-3 | "追溯太麻烦了，影响效率" | 追溯的投入在变更出现问题时会成倍收回。客开场景中，无法追溯的变更意味着无法评估风险，这是不可接受的。 |
| R6-4 | "我记得改了什么，不需要记录" | 记忆不可靠，且不可传递。追溯信息不仅服务于当前开发者，也服务于未来的维护者和其他团队成员。 |

## 违反处理机制

### 违反分级

| 级别 | 定义 | 处理方式 |
|---|---|---|
| **P0 - 严重违反** | 违反铁律的核心要求，直接进入开发或交付 | 立即停止所有操作，回退到最近的安全状态，重新从违反的铁律对应阶段开始 |
| **P1 - 显著违反** | 部分遵守铁律，但有关键环节缺失 | 暂停当前操作，补充缺失环节，经确认后可继续 |
| **P2 - 轻微违反** | 遵守了铁律精神，但形式上不完整 | 记录违反项，在当前阶段结束前补充完善 |

### 处理流程

```
检测到违反
    │
    ▼
判断违反级别 (P0 / P1 / P2)
    │
    ├──→ P0: 立即停止 + 回退 + 重新开始
    │
    ├──→ P1: 暂停 + 补充 + 确认后继续
    │
    └──→ P2: 记录 + 阶段结束前补充
```

### 回退规则

| 违反的铁律 | 回退目标 |
|---|---|
| IL-1 需求铁律 | 回退到 Phase 1（变更分析），重新澄清需求 |
| IL-2 测试铁律 | 回退到编写回归测试的步骤，不得继续代码变更 |
| IL-3 审查铁律 | 回退到审查步骤，完成审查后才能继续 |
| IL-4 验证铁律 | 回退到验证步骤，提供新鲜验证证据 |
| IL-5 文档铁律 | 回退到当前阶段开始，补充文档输出 |
| IL-6 追溯铁律 | 回退到变更记录步骤，补充追溯信息 |

### AI 自检义务

AI 在每个阶段转换点和每个关键操作前，必须**主动自检**是否违反了铁律。自检不是形式，而是实质性的检查：

1. 当前操作是否有清晰的变更需求支撑？（IL-1）
2. 是否有回归测试保护？（IL-2）
3. 变更是否经过了审查？（IL-3）
4. 是否有新鲜的验证证据？（IL-4）
5. 当前阶段的文档是否完整？（IL-5）
6. 变更代码是否可追溯到需求？（IL-6）

如果任何一条的回答是"否"，必须立即停止并处理。
