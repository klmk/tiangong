# TianGong 反模式清单 (Anti-Patterns)

> **反模式是 AI 在客开场景中最容易犯的错误。识别它们，才能避免它们。**
>
> 每条反模式都附有：编号、常见借口、核心反驳、违反的铁律、正确做法。

---

## 需求阶段反模式

### AP-01 "这个改动很小，不用分析影响"

| 项目 | 内容 |
|---|---|
| **借口** | "就改一个字段/一行配置/一个常量，影响范围很小，不需要专门分析" |
| **核心反驳** | 在已有系统中，"小改动"是事故的最高频来源。一个字段的修改可能影响数据库 schema、ORM 映射、API 响应、前端展示、报表导出等所有引用该字段的路径。影响范围不取决于改动的大小，而取决于改动被引用的广度。 |
| **违反铁律** | IL-1 需求铁律 |
| **正确做法** | 无论改动大小，都必须执行影响分析。对于小改动，可以简化分析深度，但不能跳过。至少要回答：这个改动被哪些模块引用？哪些功能依赖它？ |

---

### AP-02 "现有代码能跑就别动了"

| 项目 | 内容 |
|---|---|
| **借口** | "这段代码虽然不优雅，但一直在正常运行，最好不要动它" |
| **核心反驳** | 这个借口的问题不在于结论（不动确实比乱动好），而在于**推理过程**。"能跑"不等于"正确"，更不等于"安全"。如果变更需求要求触碰这段代码，必须先理解它的逻辑和约束，而不是因为"能跑"就回避分析。同时，如果这段代码确实是技术债务，应该在变更分析中明确记录，而不是假装它不存在。 |
| **违反铁律** | IL-1 需求铁律、IL-6 追溯铁律 |
| **正确做法** | 对现有代码保持敬畏但不回避。如果变更需要触碰"能跑"的代码，先深入理解其逻辑，评估变更风险，建立回归测试保护，然后谨慎实施。将技术债务记录在案，作为后续优化的输入。 |

---

### AP-03 "用户说的就是这个意思，不用确认"

| 项目 | 内容 |
|---|---|
| **借口** | "用户描述得很清楚，我完全理解他的意图，不需要再回头确认" |
| **核心反驳** | 客开场景中，用户的描述往往只覆盖了"期望行为"，而遗漏了"约束条件"和"边界情况"。AI 的"理解"可能遗漏了用户未提及但至关重要的隐含需求。此外，AI 的理解必须**外化**为可审查的文档，而不是停留在内部推理中。 |
| **违反铁律** | IL-1 需求铁律 |
| **正确做法** | 将理解的需求整理为结构化文档，明确列出：变更描述、影响范围、回归风险、验收标准。将文档提交给用户确认，尤其是影响范围和回归风险部分。 |

---

### AP-04 "这个需求之前做过类似的，直接照搬"

| 项目 | 内容 |
|---|---|
| **借口** | "之前做过一个类似的变更，逻辑差不多，直接参考之前的做法就行" |
| **核心反驳** | "类似"不等于"相同"。系统状态在持续变化，之前可行的方案在当前状态下可能不再适用。之前的变更可能引入了新的依赖或约束，使得"照搬"变得危险。每次变更都必须基于当前系统状态独立分析。 |
| **违反铁律** | IL-1 需求铁律、IL-3 审查铁律 |
| **正确做法** | 可以参考之前的变更作为输入，但必须基于当前系统状态重新分析影响范围和回归风险。之前的方案需要经过当前上下文的验证才能复用。 |

## 设计阶段反模式

### AP-05 "直接改就行，不用理解原有逻辑"

| 项目 | 内容 |
|---|---|
| **借口** | "需求很明确，知道要改哪里，直接改就行了，不需要花时间理解原有代码的设计意图" |
| **核心反驳** | 这是客开场景中最危险的反模式。不理解原有逻辑就动手改，相当于在不知道承重墙位置的情况下改造房屋。你改的可能是表面功能，但破坏的可能是底层约束。原有代码中的每一个"奇怪"的实现，背后可能都有一个你不知道的业务约束。 |
| **违反铁律** | IL-2 测试铁律、IL-6 追溯铁律 |
| **正确做法** | 在动手之前，必须理解变更点周围代码的设计意图、数据流和依赖关系。至少回答：这段代码为什么是这样实现的？它依赖哪些假设？修改它会影响哪些调用者？ |

---

### AP-06 "这个模块很简单，不会影响其他模块"

| 项目 | 内容 |
|---|---|
| **借口** | "这个模块功能很简单，逻辑独立，改它不会影响其他模块" |
| **核心反驳** | "简单"和"独立"是主观判断，不是客观事实。一个模块看起来简单，可能是因为你没有看到它的调用者。一个模块看起来独立，可能是因为你没有追踪它的数据依赖。影响分析必须基于实际的代码依赖关系，而不是主观判断。 |
| **违反铁律** | IL-1 需求铁律、IL-3 审查铁律 |
| **正确做法** | 通过代码分析工具或手动追踪，确认模块的实际依赖关系。列出所有调用者和被调用者，评估变更的传播路径。即使最终确认影响范围确实很小，这个确认过程也不能省略。 |

---

### AP-07 "用设计模式重构一下会更好"

| 项目 | 内容 |
|---|---|
| **借口** | "现有代码结构不好，趁这个机会用 XX 设计模式重构一下" |
| **核心反驳** | 客开场景的核心目标是**安全地满足变更需求**，而不是趁机重构。重构是好事，但在客开场景中，未经评估的重构会引入巨大的回归风险。每一次额外的变更都是额外的风险点。如果确实需要重构，应该作为独立的变更需求，经过完整的分析和评估。 |
| **违反铁律** | IL-1 需求铁律、IL-2 测试铁律 |
| **正确做法** | 聚焦于变更需求本身，用最小化的改动满足需求。如果发现现有代码结构确实需要优化，记录为技术债务，建议作为独立的变更任务处理。 |

---

### AP-08 "不需要设计文档，直接写代码"

| 项目 | 内容 |
|---|---|
| **借口** | "这个变更逻辑很直接，不需要写设计文档，直接写代码更高效" |
| **核心反驳** | 在客开场景中，设计文档的核心价值不是"设计"，而是"决策记录"。它记录了"为什么这样改"而不是"怎么改"。没有设计文档，意味着未来的维护者（包括 AI 自身）无法理解这段变更的来龙去脉，也无法评估后续变更的风险。 |
| **违反铁律** | IL-5 文档铁律 |
| **正确做法** | 即使是简单的变更，也要输出简要的设计文档，至少包含：变更方案、影响分析、选择此方案的理由。复杂变更则需要更详细的设计文档。 |

## 编码阶段反模式

### AP-09 "先改代码，再补回归测试"

| 项目 | 内容 |
|---|---|
| **借口** | "先改代码看效果，然后再补回归测试，这样更高效" |
| **核心反驳** | 这是与 IL-2 测试铁律直接对立的行为。先改代码意味着在没有安全网的情况下修改系统。如果改错了，你甚至不知道改坏了什么，因为你在改之前没有建立"正确行为"的基线。回归测试必须在代码变更之前编写，先捕获现有行为，再实施变更。 |
| **违反铁律** | IL-2 测试铁律 |
| **正确做法** | 严格执行"测试先行"：先分析影响范围 → 编写回归测试 → 确认测试通过（捕获现有行为）→ 实施代码变更 → 确认所有测试通过。 |

---

### AP-10 "这个函数没人调用，可以随便改"

| 项目 | 内容 |
|---|---|
| **借口** | "通过搜索发现这个函数没有直接调用者，可以放心修改或删除" |
| **核心反驳** | "没有直接调用者"不等于"没有调用者"。函数可能通过反射、动态调用、配置引用、序列化等方式被间接使用。此外，函数可能是公共 API 的一部分，被外部系统依赖。删除或修改前，必须确认所有可能的调用路径。 |
| **违反铁律** | IL-1 需求铁律、IL-2 测试铁律 |
| **正确做法** | 通过多种方式确认函数的使用情况：直接调用搜索、反射调用分析、配置文件引用检查、公共 API 文档检查。如果确认可以安全修改，仍需编写回归测试作为保护。 |

---

### AP-11 "复制粘贴现有代码，改几行就行"

| 项目 | 内容 |
|---|---|
| **借口** | "现有代码里已经有类似的逻辑，复制过来改几行就能满足需求" |
| **核心反驳** | 复制粘贴是技术债务的温床。在客开场景中，复制的代码可能包含你不理解的隐含约束和业务逻辑。当原始代码被修改时，复制的代码不会同步更新，导致行为不一致。此外，复制粘贴增加了系统的维护成本和变更风险。 |
| **违反铁律** | IL-5 文档铁律、IL-6 追溯铁律 |
| **正确做法** | 优先考虑复用（提取公共方法/组件），而非复制。如果确实需要类似的逻辑，应该提取为可复用的单元，并在文档中说明复用关系。如果因时间限制必须复制，必须在变更记录中明确标注复制的来源和差异。 |

---

### AP-12 "注释掉旧代码，加上新代码就行"

| 项目 | 内容 |
|---|---|
| **借口** | "旧代码可能以后还要用，先注释掉，加上新代码，这样更安全" |
| **核心反驳** | 注释掉的代码不是"安全备份"，而是"噪音"。它会降低代码可读性，增加后续维护的认知负担。在版本控制系统中，旧代码永远不会丢失，注释掉它是多此一举。更危险的是，注释掉的代码可能被误认为是"可以恢复的逻辑"，导致混淆。 |
| **违反铁律** | IL-5 文档铁律、IL-6 追溯铁律 |
| **正确做法** | 直接删除旧代码，依靠版本控制系统保留历史。在提交信息中记录删除的原因和对应的变更需求。如果旧代码确实可能需要恢复，在变更文档中说明。 |

---

### AP-13 "改完代码记得跑一下测试"

| 项目 | 内容 |
|---|---|
| **借口** | "代码改完了，跑一下测试看看有没有问题" |
| **核心反驳** | "跑一下测试"的措辞暴露了一个问题：把测试当作事后的检查，而非事前的保护。正确的做法是：测试在代码变更之前就已经编写好并确认通过，代码变更后运行测试是为了验证变更没有破坏现有行为。如果测试在变更后才编写，你无法区分测试失败是因为代码写错了还是测试写错了。 |
| **违反铁律** | IL-2 测试铁律、IL-4 验证铁律 |
| **正确做法** | 回归测试在代码变更之前编写并确认通过 → 实施代码变更 → 运行全部测试（回归 + 新增）→ 确认所有测试通过。测试是安全网，不是事后检查。 |

## 审查阶段反模式

### AP-14 "我只改了一行，不会影响其他功能"

| 项目 | 内容 |
|---|---|
| **借口** | "变更内容只有一行代码，不可能影响其他功能" |
| **核心反驳** | 一行代码的修改可能影响整个系统的行为。关键不在于改了多少行，而在于改了什么。修改一个全局配置、一个共享状态、一个公共方法的返回值，都可能产生级联影响。审查的重点是变更的**影响范围**，而非变更的**代码量**。 |
| **违反铁律** | IL-3 审查铁律 |
| **正确做法** | 无论变更大小，都按审查维度逐项检查：变更必要性、影响完整性、回归安全性、向后兼容性、数据一致性、边界条件。对于一行代码的变更，审查可以更快速，但不能跳过。 |

---

### AP-15 "手动测试过了，没问题"

| 项目 | 内容 |
|---|---|
| **借口** | "我在本地手动测试了一下，功能正常，没有发现问题" |
| **核心反驳** | 手动测试存在根本性缺陷：不可重复、不可自动化、覆盖面有限。你手动测试的场景只是众多可能场景中的一小部分。客开场景中，回归测试必须覆盖所有受影响的功能路径，这是手动测试无法做到的。手动测试可以作为补充，但不能替代自动化回归测试。 |
| **违反铁律** | IL-2 测试铁律、IL-4 验证铁律 |
| **正确做法** | 手动测试可以作为快速验证手段，但最终验证必须依赖自动化回归测试。审查时，必须确认自动化测试的覆盖范围是否充分，而非依赖手动测试结果。 |

---

### AP-16 "代码风格不重要，功能正确就行"

| 项目 | 内容 |
|---|---|
| **借口** | "客开场景重点是功能正确，代码风格不一致无所谓" |
| **核心反驳** | 代码风格不是审美问题，而是可维护性问题。在已有系统中保持一致的代码风格，能降低后续变更的认知负担，减少因风格不一致导致的理解偏差。更重要的是，风格不一致往往意味着对现有代码缺乏理解 -- 如果你理解了现有代码的风格，自然会保持一致。 |
| **违反铁律** | IL-3 审查铁律、IL-5 文档铁律 |
| **正确做法** | 变更代码应遵循现有代码的风格约定。如果现有代码风格不统一，在变更范围内保持内部一致即可。将风格问题记录为技术债务，建议统一处理。 |

---

### AP-17 "这个变更和上次的审查结论一样，不用重新审查"

| 项目 | 内容 |
|---|---|
| **借口** | "这次变更和上次几乎一样，上次已经审查过了，这次不需要重新审查" |
| **核心反驳** | 系统状态在持续变化。上次的审查结论基于上次的系统状态，当前状态下可能有新的依赖、新的约束、新的风险。每次变更都必须独立审查，不能因为"类似"就跳过。审查的价值在于发现当前变更在当前系统状态下的风险，而非确认"这次和上次一样"。 |
| **违反铁律** | IL-3 审查铁律 |
| **正确做法** | 每次变更都执行独立的审查流程。可以参考上次的审查结论作为输入，但必须基于当前系统状态重新评估。重点检查：自上次变更以来，系统是否有新的变化可能影响本次变更？ |
