# AI错题本模板（客开版）

> 本模板用于记录AI在客开变更过程中犯的错误，分析根因，制定补强规则。
> 客开版在 Paoding 基础上增加了客开场景特有的错误分类。

---

## 使用说明

1. 每次变更完成后，梳理本次变更中AI犯的所有错误
2. 按错误分类归档
3. 每个错误必须完成根因分析和补强规则制定
4. 定期回顾错题本，识别重复出现的模式

---

## 错误分类体系

### 通用错误分类（继承自 Paoding）

| 编号 | 错误类型 | 说明 |
|------|----------|------|
| G-01 | 需求理解错误 | 误解了用户的需求意图 |
| G-02 | 设计错误 | 方案设计存在缺陷 |
| G-03 | 编码错误 | 代码实现存在Bug |
| G-04 | 测试错误 | 测试用例设计或实现有误 |
| G-05 | 性能错误 | 引入了性能问题 |
| G-06 | 安全错误 | 引入了安全漏洞 |

### 客开特有错误分类

| 编号 | 错误类型 | 说明 |
|------|----------|------|
| C-01 | 代码理解错误 | 误解了现有代码的逻辑或用途 |
| C-02 | 影响评估错误 | 低估了变更的影响范围 |
| C-03 | 兼容性错误 | 变更与现有架构不兼容 |
| C-04 | 回归测试遗漏 | 漏掉了应该回归测试的功能 |
| C-05 | 数据兼容性错误 | 变更导致数据格式或结构不兼容 |

## 附录：根因分类参考

### 知识缺失型

**特征：** AI缺少必要的领域知识或系统知识，导致做出错误判断。

**补强方向：**
- 在变更前补充必要的领域知识
- 更新系统知识库，确保AI可以获取最新的系统知识
- 在 prompt 中提供必要的背景信息

### 理解偏差型

**特征：** AI误解了需求或代码意图，方向正确但理解有偏差。

**补强方向：**
- 加强确认环节，关键理解必须与使用者确认
- 增加代码阅读的深度和广度
- 使用多种方式交叉验证理解是否正确

### 惯性思维型

**特征：** AI用从零开发的思维处理客开问题，忽略了客开的特殊约束。

**补强方向：**
- 强化客开思维模型的训练
- 在关键决策点增加客开视角的检查
- 明确区分"新建"和"修改"的不同处理方式

### 注意力缺失型

**特征：** AI遗漏了重要的细节或约束，导致错误。

**补强方向：**
- 使用检查清单确保不遗漏关键步骤
- 增加自我审查的深度
- 对关键信息进行显式标注和追踪

### 能力边界型

**特征：** 超出了AI当前能力范围，需要人工介入。

**补强方向：**
- 更新AI能力画像，明确标注能力边界
- 在超出能力范围时主动请求人工帮助
- 逐步扩展能力边界，但不过度承诺
